# 2026-06-10 AI 知识体系学习记录

今天围绕 [[concepts/上下文工程]]、eval 基线、[[concepts/RAG与Memory边界|RAG 与 Memory 边界]] 做了一轮概念澄清。

## 核心理解

1. **上下文工程应该先有 eval 基线**
   不是先写一大段 prompt，而是先建立一组能代表真实任务的 eval case。没有 eval，就无法判断 prompt、context、RAG 或工具说明是否真的变好。

2. **eval 基线不需要一开始很大**
   起步可以是 20-50 条真实任务样本，覆盖常见任务、失败案例、边界场景和高风险场景。重点是有输入、期望行为、评分 rubric 和当前系统 baseline。

3. **上下文工程不只是 prompt engineering**
   Prompt engineering 更像“写好指令”；上下文工程是管理模型推理时看到的全部高信号信息，包括任务目标、规则、用户输入、外部知识、示例、工具、记忆、历史和 token 管理策略。

4. **上下文工程的核心目标**
   找到完成任务所需的最小高信号 token 集合。信息太少会让模型反复猜；信息太多会导致 [[concepts/上下文腐化]]、注意力稀释和指令漂移。

5. **上下文工程一般包含 8 部分**
   任务目标、系统规则、输入整理、外部知识、示例模板、工具上下文、记忆历史、上下文管理策略。

6. **RAG 与 Memory 的边界**
   RAG 解决“答案依据在哪里”，面向外部知识、文档、政策、网页、代码库。Memory 解决“这个用户 / agent / 组织之前发生过什么”，面向偏好、历史决策、任务状态、失败经验和长期连续性。

7. **Memory 维护主体连续性**
   主体可以是用户、agent 或组织。Memory 让 AI 不再每次都像第一次见面，而是能记得偏好、目标、历史、经验和已完成状态。但 Memory 必须带来源、时间、权限、有效性和冲突处理，旧记忆不能自动压过当前事实。

## 今日形成的判断句

**Eval 基线是上下文工程的起点；上下文工程是把模型完成任务所需的高信号信息组织进推理现场；RAG 提供外部知识，Memory 维持主体在时间中的连续性。**
