# 2026-06-09 AI 知识体系学习记录

今天的学习重点不是新增很多概念，而是把 [[domains/AI知识体系]] 的结构从“知道有 8 层”推进到“能把层级串成一个系统生命周期”。

## 今日学到的主线

AI 系统可以按一条生成链理解：

`基础理论 -> 模型构建 -> 推理能力 -> 上下文与知识 -> Agent 系统 -> 评估可靠性 -> 产品组织 -> 自我改进`

这条链不是分类表，而是系统如何从“能学”走到“能在真实产品里持续改进”的路径。第 8 层不是孤立终点，而是反馈环：它把真实运行中的 trace、失败、用户反馈和产品结果，反哺第 6 层 eval、第 4 层上下文/知识，以及证据充分时的第 2 层模型构建。

## multi-agent 与 subagent

[[concepts/AgentHarness与TraceEval]] 的理解更清楚了：

- **subagent** 是 delegation 的结果：主 agent 或 harness 把一个局部任务派给子执行单元。
- **multi-agent** 是 orchestration 的系统形态：多个 agent 可以主从协作、同级协作、评审、辩论或流水线分工。
- 两者主归属都是第 5 层 Agent 系统层，但关注点不同：subagent 关注任务拆分和结果回收，multi-agent 关注多个 agent 的协调结构、共享状态、通信和责任归因。

稳定判断：`delegation -> subagent`；`orchestration -> multi-agent system`。

## 第 3 层与第 4 层

第 3 层和第 4 层的关系可以这样记：

- 第 3 层是 **运行时能力边界**：模型在这次请求里怎么推理、用多少 test-time compute、选择哪个模型或 effort、能否满足 serving 成本和延迟。
- 第 4 层是 **任务信息供应系统**：这次任务要给模型什么 prompt、context、RAG 文档、memory、skill 和工具描述。

一次请求的数据流通常是 `第 4 层准备输入 -> 第 3 层消化输入并产生能力表现`；但体系递进上，第 3 层先说明模型运行时有什么能力，第 4 层再说明如何把任务现场接入这些能力。

稳定判断：第 4 层决定模型看到什么，第 3 层决定模型怎么处理、处理到什么水平、花多少成本。

## 1 到 8 层的整体串联

8 层可以用一句话复述：

> 先解释模型为什么能学，再训练出模型；运行时发挥能力；给它接入任务知识；让它像 agent 一样行动；用 eval 判断是否可靠；放进真实产品和组织流程；最后用真实反馈改进前面的层。

分层理解：

| 层级 | 今天形成的理解 |
|---|---|
| 1 基础理论层 | 解释“为什么能学”，提供架构、优化和 scaling 直觉 |
| 2 模型构建层 | 把数据、反馈和偏好写进模型权重 |
| 3 推理与能力层 | 权重在运行时表现为 reasoning、多模态、serving、routing |
| 4 上下文与知识层 | 把当前任务现场、外部知识、历史和程序性知识喂给模型 |
| 5 Agent 系统层 | 让模型进入 loop，调用工具、保持状态、执行多步任务 |
| 6 评估与可靠性层 | 消费答案、trace、成本、风险和最终状态，判断是否可靠 |
| 7 产品与组织层 | 把可靠能力嵌入真实工作流、权限、发布和数据飞轮 |
| 8 自我改进与前沿层 | 把真实反馈变成可审查改动，反哺第 6、4、2 层 |

## 可视化理解

[[canvases/ai-knowledge-system.canvas]] 改成自底而上的竖向 ladder 后，更符合学习直觉：

- 第 1 层在底部，表示理论地基。
- 第 2/3 层向上生成模型能力。
- 第 4/5 层把能力接入任务并变成行动。
- 第 6/7 层把行动变成可靠产品和组织价值。
- 第 8 层在顶部，但右侧回流到第 6、4、2 层，表示自我改进是反馈环。

这个图的作用不是替代表格，而是在概念混乱时提供一个“系统位置感”。

## 下次学习时的使用方法

遇到新概念时先写一行：

`概念名 -> 主层级 -> 上游输入 -> 下游输出 -> 评估方式 -> 可能反哺`

如果这一行写不出来，说明还没有真正理解它在 AI 系统里负责哪一环。下一步可以用这个模板练习：`context compression`、`dynamic workflow`、`subagent`、`model routing`、`eval harness`。
