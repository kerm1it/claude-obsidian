# 2026-06-14 AI 知识体系学习记录

今天围绕 agent 运行时的短期状态、trace 生成、Codex / Claude Code 落地方式和 grader 概念做了一轮澄清。主线是：**scratchpad 是当前任务工作台；harness 通过 runtime 边界生成 trace；session transcript 是原始转录，hooks 才适合生成结构化 trace；grader 把 trace / 输出变成可用评分信号。**

## 核心理解

1. **Scratchpad 是当前任务的临时工作台**
   Scratchpad 用来记录当前任务目标、已完成步骤、关键发现、TODO、当前假设、工具结果和下一步动作。它属于 short-term memory / context engineering，不等同于长期 Memory，也不是 RAG。

2. **Scratchpad 的内容不一定长期保存**
   当前任务里的 checklist、下一步计划、临时假设通常任务结束即可丢弃。只有反复出现、可复用或对未来任务有价值的内容，才可能被巩固成 Memory、Skill、Eval case 或 wiki note。

3. **Trace 是 harness 在运行时边界打点生成的结构化轨迹**
   Trace 不是 agent 主动写日记，也不一定是模型内部推理链；它主要由 runner / harness 在 model call、tool call、permission、guardrail、handoff、error、retry、final state 等可观测边界记录。

4. **Trace 的核心目的是可复盘、可评分、可审计**
   一条有效 trace 至少要回答：agent 接到什么任务、看到了哪些上下文、调用了哪些工具、每个工具调用为什么被允许或拒绝、最终结果是什么、失败发生在哪一步。

5. **落地时不要只依赖 session transcript**
   Codex、Claude Code 的 session transcript 适合个人复盘和原始证据保存，但不适合作为长期稳定的 eval 数据接口。它通常太全文、含隐私、格式可能变化，也不方便直接统计工具调用、权限判断和失败原因。

6. **Hooks 更适合生成结构化 trace**
   对 Codex / Claude Code 这类现成 coding agent，最佳做法是保留 session transcript 作为原始材料，同时用 hooks 在 `SessionStart`、`UserPromptSubmit`、`PreToolUse`、`PermissionRequest`、`PostToolUse`、`SubagentStart/Stop`、`PreCompact/PostCompact`、`Stop/SessionEnd` 等生命周期点抽取结构化事件，写入 `traces/*.jsonl`。

7. **Hook trace 应保存摘要和原文指针**
   长期保存时不应无脑保存完整 prompt、用户内容、工具输出和 secret。更稳的结构是：hook trace 保存事件类型、工具名、参数摘要、权限决策、结果摘要、状态、错误和 `transcript_path`；需要深挖时再回读原始 session。

8. **Grader 是 eval 里的评分逻辑**
   Grader 负责把模型输出、agent trace、工具调用结果或最终状态转成结构化信号：score、pass/fail、label、failure reason、uncertainty。它不负责产生答案，而负责判断答案或过程是否可靠。

9. **Grader 类型要按任务选择**
   可确定任务优先用 deterministic / code grader，例如测试、schema、状态检查；开放式语义任务才用 LLM judge；高风险或模糊任务需要 human grader 或 human gold 校准。LLM judge 是 grader 的一种，不是绝对裁判。

10. **Agent eval 的稳定链路**
    `session / hooks -> trace -> grader -> score / failure reason -> regression eval / memory / skill / harness fix`。失败 trace 可以沉淀为回归 eval，重复成功或失败模式可以进入 skill / memory / self-improvement 候选，但必须经过第 6 层质量门。

## 落地判断

| 场景 | 首选做法 |
|---|---|
| 个人复盘 | 直接读 session transcript |
| wiki / memory 总结 | session transcript + Stop / SessionEnd hook 摘要 |
| agent eval | hooks 生成结构化 trace，再由 grader 消费 |
| 权限审计 | PreToolUse / PermissionRequest / PostToolUse hook 必须记录 |
| 失败回归集 | hooks 定位失败步骤，必要时回读 session 补上下文 |
| 长期数据资产 | 保存脱敏摘要、artifact id、transcript_path 和 lineage，不默认保存全部原文 |

## 今日形成的判断句

**Scratchpad 是当前任务的临时工作台；trace 是 harness 在运行时边界生成的结构化证据；Codex / Claude Code 里 session 适合复盘，hooks 适合 eval 级 trace；grader 是把输出或 trace 变成分数、标签和失败原因的评分逻辑。**

## 下一步

- 写一个最小 `traces/*.jsonl -> eval_cases/*.yaml` 的转换模板，把失败 trace 变成 regression eval。
- 继续区分 `grader`、`judge`、`verifier`、`monitor`、`reward model` 在第 6 层中的角色。
