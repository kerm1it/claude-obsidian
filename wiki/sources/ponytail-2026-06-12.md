## 一句话定位

> "Long ponytail. Oval glasses. Has been at the company longer than the version control."

让 AI 编码助手像"最懒的资深开发者"一样思考——系统性地优先使用已有方案而非编写新代码。声称结果：减少 80-94% 代码、3-6× 更快、42-75% 更便宜。

---

## 核心机制：决策阶梯

Agent 在写代码前暂停，沿着阶梯逐步上升，停在第一个适用的阶梯：

| 阶梯 | 行动 | 原则 |
|------|------|------|
| 1 | 这真需要存在吗？→ 跳过 | YAGNI |
| 2 | 标准库已有？→ 用它 | stdlib first |
| 3 | 原生平台特性？→ 用它 | platform native |
| 4 | 已安装依赖覆盖？→ 用它 | dep before code |
| 5 | 一行代码就够？→ 写一行 | one-liner |
| 6 | 只有此时：写最小可行实现 | minimal |

**不削减的底线**：验证、错误处理、安全、可访问性永不被 skip。每个快捷方式用 `ponytail:` 注释标注升级路径。

## 关键洞察

1. **代码最小化是 Agent 行为模式，不是功能**：Ponytail 不是 linter 或 dead code detector——它是一个 prompt-layer behavior constraint，让 Agent 天生倾向少写代码
2. **决策阶梯 = 系统化常识**：梯子将"有经验的开发者不写不必要的代码"这一隐性直觉编码为显式 check-list
3. **ponytail: 注释 = 技术债的有意识管理**：不禁止快捷方式，但要求标记升级路径，让债可见、可追踪
4. **成本降低是副产品**：Ponytail 的 pitch 是"写更少的代码 = 更好的代码"，token/成本降低只是结果
5. **与 Skill 互补**：Skill 告诉 Agent *怎么做*一类任务，Ponytail 告诉 Agent *先别急着写*，两者叠加产生更精确的输出
