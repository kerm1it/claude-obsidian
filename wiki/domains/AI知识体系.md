## 第一轮回填：微调 / RAG / 上下文工程

[[concepts/微调RAG上下文工程选择框架]] 将第 2 层和第 4 层连接起来：

- **上下文工程**：先把任务目标、约束、工具、历史和示例组织清楚；这是最低成本的第一步。
- **RAG**：当问题来自外部、私有、频繁变化或需要出处的知识时，把知识留在模型外部，通过检索进入上下文。
- **微调 / PEFT**：当问题是稳定行为、格式、风格、工具调用、边界案例、成本或延迟时，再考虑改权重或 adapter。
- **Eval**：三者都必须经过第 6 层验证；没有 eval，无法判断优化是否真实改善。

判断顺序：`eval 基线 → 上下文工程 → RAG 补知识 → 微调稳行为 → 蒸馏/小模型降成本`。

## 第二轮回填：RAG 与 Memory 边界

[[concepts/RAG与Memory边界]] 将第 4 层拆成两个稳定职责：

- **RAG**：外部知识进入上下文的机制，重点是出处、更新、检索质量和 groundedness。
- **Memory**：agent / 用户 / 组织跨任务连续性的机制，重点是写入、巩固、衰减、冲突解决、权限和 stale guard。
- **短期上下文**：当前任务 scratchpad、checklist、tool result，不等同于长期 memory。
- **Skill**：当 memory 中的经验变成可复用程序或规则时，应升级为 skill，进入第 4 层并服务第 5 层 Agent。

稳定判断：`外部事实 → RAG`；`交互历史/偏好/经验 → Memory`；`当前任务临时状态 → short-term context`；`可复用执行程序 → skill`。

这一轮把第 4 层和第 8 层连得更清楚：Dream/self-improvement 的一个实际入口是把 trace 和失败案例巩固为 memory，再把稳定经验沉淀为 skill、eval 或知识库更新。

## 第三轮回填：Agent harness 与 trace/eval

[[concepts/AgentHarness与TraceEval]] 将第 5 层和第 6 层接起来：

- **Harness**：agent 的运行时层，管理 runner loop、tool registry、state/session、workspace/sandbox、权限、handoff、guardrail 和 trace。
- **Trace**：harness 产出的可审计轨迹，记录 generation、tool call、handoff、guardrail、error 和最终产物。
- **Eval**：消费 trace、最终状态和成本数据，判断行为可靠性，并把失败归因回 prompt、tool、memory、model 或产品流程。
- **稳定边界**：harness 让 agent 可运行且可观测；eval 让 agent 可判断且可改进。

稳定判断：`prompt 是 harness 输入`；`tool use 是 harness 管理的动作`；`workflow 是 harness 可执行的固定路径`；`trace/eval 是 harness 进入可靠性层的接口`。

## 第四轮回填：Dream 与 self-improvement

[[concepts/Dream与SelfImprovement工程原语]] 将第 8 层落到工程闭环：

- **Dream / consolidation**：读取 memory store、session、trace，生成可审查的 memory diff。
- **Reflection**：把失败反馈转成语言经验，进入 episodic memory 或 skill。
- **System improvement**：改 context、tools、harness、eval、docs 或 workflow。
- **Weight-level self-adaptation**：改 adapter / fine-tune / weights，属于高风险前沿层，必须有 hold-out eval 和人工门。

稳定链路：`trace → eval/outcome → dream/reflection → proposed diff → gate → apply → 下一轮 trace`。

这一轮把第 8 层明确成反馈环，不是第 9 层：成熟的 self-improvement 先反哺第 4 层 memory/context、第 5 层 harness/tool、第 6 层 eval；只有在证据充分时才反哺第 2 层模型构建。

## 第五轮回填：Skill 与 tool/memory/workflow 边界

[[concepts/Skill与ToolMemory边界]] 将第 4 层程序性知识与第 5 层执行动作区分开：

- **Skill**：可版本化程序性知识包，包含 `SKILL.md`、scripts、references、assets 和触发 metadata。它回答“这一类任务应该怎么做”。
- **Tool**：可执行动作或 API。它回答“系统能做什么动作”。
- **Memory**：跨任务保留事实、偏好、经验和历史。稳定经验反复出现后，可以编译成 skill。
- **Workflow**：较固定的步骤编排。Skill 可以调用 workflow，也可以指导开放式任务。
- **Eval / security gate**：判断 skill 是否真的提高成功率，是否带来路由噪声、token overhead、权限扩大或供应链风险。

稳定链路：`trace / repeated workflow → reflection or curation → candidate skill → eval / scan → install → harness route and execute`。

这一轮把第 4、5、6、8 层接得更细：第 8 层从失败 trace 或 memory 中发现可复用模式，第 4 层把它沉淀为 skill，第 5 层 harness 路由并执行，第 6 层用任务成功率、调用准确率和安全扫描决定是否保留。

## 第六轮回填：推理时计算与 reasoning

[[concepts/TestTimeCompute]] 将第 3 层“推理与能力层”和第 5 层 Agent loop 接起来：

- **Test-time compute**：模型权重不变时，在推理阶段投入更多 token、采样、搜索、verifier、reranker 或 tool loop 来换质量、可靠性和可监控性。
- **Reasoning**：模型运行时的推理策略，不等同于 prompt；prompt 给任务信息，reasoning 决定如何处理这些信息。
- **Effort / budget**：产品化控制面，用于表达质量、成本、延迟之间的取舍。
- **Verifier / process supervision**：第 6 层可靠性机制，判断多条 reasoning path、搜索节点或最终答案。
- **CoT monitorability**：把推理过程本身变成可观测对象，但不能把 CoT 当成绝对真实的内心独白。

稳定链路：`模型权重 → reasoning policy / effort → search or thinking or sampling → tool/result feedback → verifier/eval → final answer/action`。

这一轮明确第 3 层如何进入 agent：TTC 在单轮问答里表现为更多 thinking / sampling，在 agent 中表现为更长 trace、更多 tool call、更复杂的计划修正。第 6 层必须衡量成功率、成本、延迟、trace 可监控性和失败恢复，而不只看答案是否漂亮。

## 第七轮回填：多模态与世界模型边界

[[concepts/多模态与世界模型边界]] 将第 3 层的“感知/生成能力”和“动态模拟能力”拆开：

- **多模态**：模型能接收、融合、生成或转换文本、图像、音频、视频、3D、传感器和动作等信号。
- **世界模型**：模型能表示环境状态、时间动态、对象持久性、物理约束、行动后果和反事实轨迹。
- **原生多模态**：多种模态由同一个模型处理，区别于转录/识别/合成 pipeline。
- **多模态长上下文**：视频、音频、文档等被纳入第 4 层上下文工程和检索。
- **VLA / embodied agent**：当多模态能力输出 action 或 control，它进入第 5 层 Agent 系统。

稳定判断：`能处理多种信号 → 多模态`；`能预测行动和时间导致的状态变化 → 世界模型`；`能把 observation + instruction 转成 action → VLA / embodied agent`。

这一轮把第 3 层多模态能力接到真实产品：进入语音/视频助手时连接第 7 层体验和延迟；进入机器人或仿真环境时连接第 5 层行动 loop；进入物理/动态任务时必须由第 6 层评估对象持久性、行动后果、物理一致性和安全边界。

## 第八轮回填：Serving / 成本 / 延迟边界

[[concepts/Serving成本延迟边界]] 将第 3 层推理能力与第 7 层产品经济性接起来：

- **Serving**：把模型推理变成满足 SLO 的可用服务，核心是 request/context → prefill → decode → streaming/postprocess。
- **Prefill / decode**：输入 token 处理和输出 token 生成是不同性能阶段，分别影响 TTFT、TPOT 和吞吐。
- **KV cache / paged attention**：显存管理决定并发、吞吐和长上下文成本。
- **Batching / continuous batching**：提升 GPU 利用率和吞吐，但要和单请求延迟 SLO 平衡。
- **Prompt caching**：把稳定长前缀从每次重复处理变成缓存命中，是第 4 层上下文工程进入第 3/7 层的接口。
- **Cost/success**：Agent 时代不能只看 cost/request，要看完成一个任务或 workflow 的真实成本。

稳定链路：`产品 SLO → eval 基线 → 选模型/减 token → cache → batching/KV/prefill-decode 优化 → 蒸馏/量化/小模型 → cost/success 监控`。

这一轮把“能力能不能用”从模型榜单拉回产品现实：第 3 层能力只有经过第 6 层监控 TTFT、TPOT、P95/P99、goodput、error rate 和 cost/success，并满足第 7 层价格、体验和并发约束，才算真的可部署。

## 第九轮回填：Skill library routing / lifecycle

[[concepts/SkillLibraryRouting生命周期]] 将第 4 层单个 skill 扩展为库级程序性知识治理：

- **Library routing**：不是把所有 skills 塞进上下文，而是先检索候选、再选择加载、最后按需读取 references/scripts/assets。
- **Metadata surface**：`name`、`description`、tags 和 requirements 是 routing surface，也是攻击面。
- **Lifecycle**：collection/create → normalize → index/retrieve → select/load → execute → attribute outcome → update/deprecate/retire。
- **Evidence-gated update**：从 trace 生成或更新 skill 时，必须有成功证据、失败归因、回归测试和安全审查。
- **Retirement**：重复、过时、环境依赖强或负增益 skill 必须能合并或退役，否则 library 会污染未来上下文。

稳定链路：`trace / human skill → metadata normalization → retrieval → selection → harness execution → outcome attribution → evidence-gated update or retirement`。

这一轮把第 4 层和第 6/8 层接得更紧：skill library 是可演化知识库，但每次演化都要被第 6 层 eval/security gate 审查；第 8 层 self-improvement 只能提出候选改动，不能无证据污染 library。

## 第十轮回填：Reasoning eval / verifier

[[concepts/ReasoningEvalVerifier边界]] 将第 3 层 reasoning 和第 6 层可靠性正式接起来：

- **Outcome verifier / ORM**：评分最终答案或最终状态，适合 best-of-N、回归 eval 和训练 reward。
- **Process verifier / PRM**：评分推理步骤或 progress，适合 search pruning、过程监督和 dense reward。
- **Trace grader**：评分完整 agent trajectory，检查工具调用、恢复路径、风险行为、成本和最终状态。
- **CoT monitor**：监控暴露 reasoning 中的风险信号，但不能把 CoT 当成绝对真实的内心过程。
- **Reward model**：训练出的 scorer，不是天然可信裁判；它本身也需要独立 eval、校准和 hold-out 检查。

稳定链路：`reasoning candidates / trace → verifier / grader / monitor → score / label / failure reason → rerank / gate / train / improve`。

这一轮把第 6 层变成更具体的“质量接口”：它消费第 3 层推理时计算和第 5 层 harness trace，反哺第 2 层训练、第 4 层上下文和 memory、第 5 层工具/harness，以及第 8 层 self-improvement。核心风险是 reward hacking、grader drift、benchmark leakage 和过程监督泛化不足。

## 第十一轮回填：Multimodal eval / simulation eval

[[concepts/MultimodalEvalSimulationEval边界]] 将第 3 层多模态/世界模型能力接到第 6 层评估：

- **静态多模态 eval**：评估图像、图表、文档、地图、化学结构等跨视觉/语言理解。
- **视觉数学/科学推理 eval**：区分感知错误、图形理解错误和抽象推理错误。
- **视频/音频 eval**：评估时序理解、长视频召回、音频/字幕/画面融合和事件定位。
- **多模态生成安全 eval**：评估语音、图像、视频生成的身份、隐私、版权、安全和人格化风险。
- **World-model eval**：评估物理一致性、对象持久性、action-conditioned rollout 和 controllability。
- **Embodied / simulation eval**：在环境中评估任务成功、碰撞、低层动作、恢复和跨环境泛化。

稳定链路：`modality inputs / generated media / action rollout → benchmark + human/judge + simulator state checks → score / violation / failure reason → product gate / training data / harness fix`。

这一轮明确：多模态模型的“看见/听见/生成”属于第 3 层，但“它到底可靠吗”属于第 6 层；当输出进入行动或仿真环境时，最终状态、物理违反和安全边界比视觉偏好更重要。

## 第十二轮回填：Product / organization layer

[[concepts/ProductOrganizationLayer边界]] 将第 6 层可靠性与第 7 层产品组织价值正式接起来：

- **Product / organization layer**：把已经验证的 AI 能力嵌入产品表面、业务工作流、组织角色、单位经济和反馈数据中。
- **Workflow fit**：第 7 层从真实痛点和可重复工作流出发，不从“模型能做什么”的炫技出发。
- **Human role / approval boundary**：定义哪些动作可自动执行，哪些需要审批、接管、回滚或责任人。
- **Telemetry / data flywheel**：记录接受、拒绝、失败、人工覆盖、成本、延迟和 ROI，再回填 eval、context、skills、tools 或训练数据。
- **Company brain / software factory**：组织知识和软件流程被重写为 AI 可查询、可执行、可评估的闭环。

稳定链路：`validated capability + SLO + unit economics → product surface / workflow integration → human role / approval boundary → telemetry / data rights → eval and roadmap → self-improvement feed`。

这一轮明确：第 7 层不是商业杂项，也不是 UI 层；它是 AI 系统进入真实组织后的接口层。第 6 层说“这个系统是否可靠”，第 7 层说“它是否在真实工作流中持续创造价值”，第 8 层再把这些反馈变成可审查改动。

## 第十三轮回填：Model routing / mixture

[[concepts/ModelRoutingMixture边界]] 将第 3 层推理运行时与第 6/7 层质量和产品约束接起来：

- **Runtime model routing**：在请求进入时，按任务、风险、SLO、成本、上下文和 eval 历史选择模型、effort、cascade 或 ensemble。
- **Internal MoE**：模型内部选择专家参数，主属第 2 层模型构建，不等同于应用层 router。
- **Cascade / escalation**：便宜模型先处理，低置信、失败或高风险时升级到强模型或人工门。
- **Mixture-of-Agents / ensemble**：多个模型或 agents 并行生成、互相读取或聚合，偏质量提升，成本和延迟更高。
- **Compound system model allocation**：在多模块 agent/workflow 中给不同模块分配模型，跨到第 5 层 harness/orchestration。

稳定链路：`request / task features / risk / SLO → routing policy or learned router → cheap-fast default or strong model → confidence / verifier / guardrail → escalate, parallelize, or stop → log quality + cost + latency → update eval/router/model pool`。

这一轮明确：model routing 不是新的顶层，也不是简单“多接几个模型”。它主属第 3 层 inference control，但必须由第 6 层评估 cost-quality frontier、calibration、oracle regret、false downgrade 和 drift，并由第 7 层产品 SLO、用户等级、风险边界和单位经济约束。

## 第十四轮回填：Reward hacking / eval overfitting

[[concepts/RewardHackingEvalOverfitting边界]] 将第 6 层质量门的失效模式单独固定下来：

- **Reward hacking**：系统优化 reward 或 grader 漏洞，而不是优化真实目标。
- **Specification gaming**：满足任务字面规格，违背设计者真实意图。
- **Eval overfitting**：固定 eval / benchmark 分数上升，但真实分布没有改善或变差。
- **Benchmark leakage**：测试集进入训练、调参或模型选择，导致榜单分数虚高。
- **Reward tampering**：agent 修改 reward、测试、日志或监督通道本身。
- **KPI gaming**：第 7 层产品组织优化代理指标，牺牲真实用户价值。

稳定链路：`proxy metric / reward / eval → optimization pressure → shortcut / exploit / leakage / tampering → score improves while true outcome degrades → holdout eval / adversarial eval / trace monitor / human audit → revise metric, grader, environment, permissions, feedback loop`。

这一轮明确：reward hacking 不是单独的“安全层”，而是第 6 层 eval/reward/benchmark 在优化压力下的核心失效模式。它会污染第 2 层训练、第 3 层 model routing、第 5 层 agent trace、第 7 层 KPI 和第 8 层自我改进，所以所有反馈闭环都必须有 hold-out、权限、人工校准和 anti-gaming 设计。

## 第十五轮回填：Data flywheel / feedback loop

[[concepts/DataFlywheelFeedbackLoop边界]] 将第 7 层真实使用反馈正式接到第 2/4/6/8 层：

- **Data flywheel**：真实使用产生行为信号、失败 trace、人工修正、显式反馈和业务结果；系统改进后获得更多、更高质量信号。
- **Signal routing**：不是所有反馈都进训练。失败进 eval，事实缺口进 RAG，偏好进 memory，重复流程进 skill，稳定行为偏差才进入 fine-tune / distill 候选。
- **Provenance / consent**：每条信号必须保留来源、时间、权限、模型版本、prompt/tool 上下文和用途边界。
- **Anti-gaming**：反馈闭环会放大低质量标签、KPI gaming、reward hacking 和 benchmark leakage，必须经过第 6 层质量门。
- **Continuous improvement**：第 8 层 self-improvement 使用 data flywheel 证据生成 diff，但不能绕过 eval、人工门和 rollback。

稳定链路：`real usage / feedback / trace → consent + logging boundary → signal extraction and attribution → routing by evidence type → eval / context / memory / skill / tool / router / training update → hold-out validation and anti-gaming checks → product improvement`。

这一轮明确：data flywheel 主属第 7 层，不是“数据层”或“训练层”。它的核心价值在于把产品反馈安全分流到正确改进路径；只有被授权、去噪、归因清楚、经过 hold-out 验证的稳定信号，才有资格进入第 2 层模型构建。

## 第十六轮回填：Data rights / privacy / consent

[[concepts/DataRightsPrivacyConsent边界]] 将第 7 层数据飞轮前面的权限闸门固定下来：

- **Purpose classification**：先区分 service、safety、analytics、memory、eval、training、vendor sharing，不能用一个“用户数据”桶处理全部用途。
- **Consent / contract / policy gate**：用户设置、组织合同、隐私政策、法规和 provider 条款共同决定数据能否进入某条路径。
- **Memory and RAG permissions**：第 4 层知识系统必须继承文档权限、租户边界、可见/可删/可关控制。
- **Eval and training split**：失败 trace 可以成为 eval 样本，但不自动获得训练授权；训练是更强用途，需要来源追踪和更高权限门槛。
- **Tenant and vendor boundary**：企业数据默认 tenant-scoped；多 provider routing 前必须检查保留期、二次使用、子处理方和地域边界。

稳定链路：`user/org data or feedback → purpose classification → consent / contract / policy check → minimization and redaction → allowed use path → retention / deletion / export controls → audit trail`。

这一轮明确：data rights / privacy / consent 主属第 7 层，不是孤立法律章节。它把真实产品数据变成可治理资源：先决定是否允许被收集、保存、检索、评估、记忆或训练，再进入第 4/6/8/2 层的改进路径。

## 第十七轮回填：Tool risk / permissioning

[[concepts/ToolRiskPermissioning边界]] 将第 5 层 agent action 的权限边界固定下来：

- **Tool risk tier**：把工具动作从纯推理、公开只读、私有只读、可逆写入、外部副作用、高影响动作到通用执行分层。
- **Permissioning is not prompting**：prompt 可以约束模型行为，但不能替代身份、scope、凭证、沙箱、审批和审计。
- **Per-tool gate**：高风险系统要在每次 tool call 前后检查，而不是只检查用户输入或最终输出。
- **Scoped authority**：agent 应使用专用、短期、最小权限身份；不要继承人类管理员凭证。
- **Action audit**：trace 必须记录 user intent、tool、arguments、policy result、approval、credential scope、result 和 rollback evidence。

稳定链路：`proposed tool call → intent and tool classification → risk tier and data-scope check → policy / permission / credential gate → human approval or sandbox → execute with scoped authority → trace, audit, rollback, eval`。

这一轮明确：tool risk / permissioning 主属第 5 层，不是第 6 层安全口号。第 5 层 harness 负责真正执行动作，因此必须内建动作准入；第 6 层负责测试越权、审批绕过、沙箱逃逸和工具组合风险；第 7 层负责组织权限、职责和审计。

## 第十八轮回填：AI Safety Ops / Governance

[[concepts/AISafetyOpsGovernance边界]] 将第 6 层风险证据正式接到第 7 层组织运行：

- **Inventory and risk classification**：先知道有哪些 AI system、用途、owner、数据流、供应商和部署状态，再谈治理。
- **Evidence to gate**：eval、red-team、monitoring 和 incident evidence 必须进入 risk register、safeguards、release gate 和 owner assignment。
- **Release and exception control**：发布不是 eval pass 的自动结果；需要 residual risk、mitigation、例外审批和 audit trail。
- **Post-market monitoring**：上线后继续监控、分级响应、修复、复盘，并把事故反哺 eval、tool permission、data rights 和产品路线。
- **Management system**：NIST AI RMF、ISO 42001、OpenAI Preparedness、Anthropic RSP、DeepMind FSF 等框架的共同稳定形态，是把风险管理变成组织流程。

稳定链路：`AI system / use case / model capability → inventory and risk classification → eval / red-team / privacy-security review → risk register and thresholds → mitigations and owner assignment → release gate or exception approval → production monitoring and incident response → periodic review`。

这一轮明确：AI Safety Ops / Governance 主属第 7 层，不是新的“治理层”。第 6 层负责产生风险证据，第 7 层负责把证据变成可执行责任、发布门、事故响应、审计和持续改进。

## 第十九轮回填：Data Quality / Label Quality

[[concepts/DataQualityLabelQuality边界]] 将第 7 层 data flywheel 的候选信号正式接到第 6 层质量门：

- **Fit-for-purpose**：同一条数据可能适合 RAG，但不适合训练；可能适合产品 analytics，但不适合 hold-out eval。
- **Label quality**：label 要正确、一致、可解释，rubric 要清楚，模糊样本要进入 review 而不是直接训练。
- **Leakage and contamination**：train、eval、hold-out、公开 benchmark、RAG cache 之间要查重和防污染。
- **Dataset documentation**：datasheet / data card / dataset card 记录来源、用途、限制、偏差、采集和标注过程。
- **Downstream impact**：数据质量不只看字段完整，还要看清洗、重标、去重后 eval 稳定性和真实任务表现是否改善。

稳定链路：`raw data / label / feedback / trace → rights and provenance check → schema / validity / freshness / dedup checks → label rubric and ambiguity review → leakage / contamination / split check → route to eval / training / RAG / memory / skill / reject → downstream impact measurement`。

这一轮明确：data quality / label quality 主属第 6 层，不是第 2 层训练数据的附属物。它决定 data flywheel 产生的候选信号能不能成为可靠证据，再分别进入 eval、training、RAG、memory、skill 或 router。

## 第二十轮回填：Multi-provider Governance

[[concepts/MultiProviderGovernance边界]] 将第 3 层 model routing、第 5 层 tool use、第 6 层 eval/monitoring 和第 7 层 data rights/governance 接到同一个 provider 控制面：

- **Provider eligibility before routing**：先按数据分类、用途、地域、retention、training use、合同、SLA 和风险等级算出 allowed provider set，再让 router 优化成本、质量和延迟。
- **Feature-level policy**：同一 provider 的 chat、responses、batch、files、grounding、tool use、managed agents、invocation logging 和 ZDR eligibility 可能有不同数据路径。
- **Data egress boundary**：prompt、文件、tool result、trace、eval 样本和训练候选都要判断能否发给某 provider、region、subprocessor 或第三方 integration。
- **Customer-owned logs matter**：供应商默认不训练不代表没有日志；客户自己启用 invocation logging、application state、grounding 或 batch 时，也会产生新的保留和审计义务。
- **Fallback is policy-bound**：provider 故障、quota、价格或 latency 变化时，fallback 不能绕过数据、地域、权限和风险边界。

稳定链路：`request / task / data class / risk tier → provider policy matrix → allowed provider / model / tool set → model or tool routing → provider-specific retention / training-use / region / SLA controls → trace / audit / cost / quality monitoring → incident / fallback / vendor review`。

这一轮明确：multi-provider governance 主属第 7 层，不是新的“供应商层”，也不是第 3 层 model routing 的别名。第 3 层负责选择路径，第 7 层负责定义哪些路径可以被选择，并把 provider 事件反哺 risk register、eval、data rights、tool permission 和产品 SLO。

## 第二十一轮回填：Agent Identity / Delegation

[[concepts/AgentIdentityDelegation边界]] 将第 5 层 agent runtime 与第 7 层组织责任主体连接起来：

- **Actor boundary**：agent 不能只表现成“某个人的 API key”或“万能服务账号”；每次动作要能区分 agent、human、service、run 和 session。
- **Delegation grant**：用户、组织或上级 agent 授权的不是“无限代理”，而是特定任务、scope、资源、时间窗、风险等级和可撤销路径。
- **Scoped credentials**：OAuth scope、MCP token、IAM role session、session tag、capability token 等只是实现手段；稳定边界是短期、最小权限、audience-bound、可审计。
- **Step-up authorization**：低风险可自动执行，高风险动作需要增量授权或 human approval，并绑定具体工具参数和资源。
- **Audit and non-repudiation**：工具动作必须记录 agent id、run id、principal、grant、policy decision、resource、result 和 trace id。

稳定链路：`human / org / service principal → delegation grant → agent or run identity → scoped short-lived credential → tool or provider request → allow / deny / step-up / approval → audit and non-repudiation → revoke / rotate / review / incident response`。

这一轮明确：agent identity / delegation 主属第 5 层，不是第 7 层治理文档，也不是第 6 层 eval。第 7 层定义责任和授权政策，第 5 层把它变成每次工具调用携带的身份与委托链，第 6 层验证它是否可追踪、最小权限、可撤销、不可被 prompt injection 或 direct-path bypass 绕过。

## 第二十二轮回填：AI Incident Response / Postmortem

[[concepts/AIIncidentResponsePostmortem边界]] 将第 6 层监控/eval/red-team 证据正式接到第 7 层事故闭环：

- **Detect and triage**：从 production monitoring、eval regression、用户报告、red-team、tool trace、provider status 和监管投诉中识别 incident / hazard。
- **Severity beyond SLO**：AI 事故严重度不能只看延迟和错误率，还要看健康、权利、数据、越权、财产、环境、合规和受影响人群。
- **Containment first**：先止血，动作包括禁用工具、回滚 prompt/model/router、暂停 memory write、切 provider、降级人工或关闭某类 autonomous action。
- **Evidence package**：保存 prompt、context、RAG 文档、memory state、model/tool/provider version、identity grant、policy decision、trace 和影响范围。
- **Blameless postmortem to regression**：复盘不是追责文档；必须把事故转成 eval、monitor、red-team case、policy test、data quality check、tool gate 或 provider review。

稳定链路：`signal / alert / report → triage and severity → incident commander and owner → containment / rollback / disable / human override → evidence preservation and stakeholder communication → root and contributing cause analysis → blameless postmortem → action items and regression conversion`。

这一轮明确：AI incident response / postmortem 主属第 7 层，是 [[concepts/AISafetyOpsGovernance边界]] 的事故闭环子系统，不是新的“事故层”。第 6 层发现和复现风险，第 7 层组织止血、沟通、取证、复盘和责任闭环，第 8 层只能在 action item 通过 gate 后接收改进任务。

## 第二十三轮回填：Synthetic Data Governance

[[concepts/SyntheticDataGovernance边界]] 将第 2 层模型构建、第 6 层质量门、第 7 层数据权利和第 8 层自我改进连接起来：

- **Candidate, not truth**：合成数据是候选证据，不是自动可训练数据，也不是自动匿名数据。
- **Use-case routing**：SFT、distillation、RLAIF、synthetic eval、simulation、RAG 摘要、memory 候选和 self-improvement data 的门槛不同。
- **Real-data anchor**：合成数据可以补长尾和生成 teacher outputs，但不能递归替代真实分布、私有 hold-out 和真实用户结果。
- **Contamination control**：训练集、synthetic eval、公开 benchmark、seed examples 和 teacher outputs 之间必须做去重、隔离和 lineage。
- **Model-collapse guard**：递归合成数据会带来分布缩窄、尾部消失和错误累积；治理要监控 synthetic-real gap 和 collapse signals。

稳定链路：`target gap / use case → seed data rights and provenance → generator model / prompt / version / sampling policy → synthetic sample generation → dedup / leakage / diversity / label / realism / safety filters → split isolation and hold-out contamination check → downstream validation on real and private evals → route to training / eval / RAG / memory / skill / simulation / reject → monitor drift, collapse, and downstream impact`。

这一轮明确：synthetic data governance 主属第 6 层，不是第 2 层训练技巧本身，也不是第 7 层合规文档。第 2 层可以消费合成训练候选，第 7 层判断 seed 和输出的权利/用途，第 8 层可以提出合成样本，但第 6 层负责决定它能不能成为证据。

## 第二十四轮回填：Dataset Lineage / Provenance Graph

[[concepts/DatasetLineageProvenanceGraph边界]] 将第 2 层训练、第 4 层知识、第 5 层 agent trace、第 6 层质量门、第 7 层治理和第 8 层自我改进接到同一个证据血缘底座：

- **Evidence graph**：记录样本、文档、label、trace、RAG chunk、memory、skill example、synthetic sample、eval case、model checkpoint、prompt、tool run 和 release decision 的来源、转换、版本和影响面。
- **Entity / activity / actor**：稳定模型来自 W3C PROV：artifact 是 entity，ingest/train/eval/tool call 是 activity，用户、模型、pipeline、agent run 或 provider 是 actor。
- **Artifact / execution / event**：ML pipeline 可借用 TFX MLMD 模型，把每次转换的输入、输出和 context 写成可查询 lineage。
- **Run / job / dataset / facet**：数据作业可借用 OpenLineage，把跨系统的 job run、dataset、schema、quality metric、column lineage 和 source version 标准化。
- **Audit and impact query**：lineage graph 支持污染检测、删除/导出影响面、事故取证、rollback、dataset card/model card 和 self-improvement diff 证据引用。

稳定链路：`artifact / sample / trace / doc / label → source and rights capture → transformation activity → versioned dataset / manifest / graph edge → route to training / eval / RAG / memory / skill / router / reject → downstream impact query → deletion / rollback / audit / incident reconstruction`。

这一轮明确：dataset lineage / provenance graph 主属第 6 层，不是新的“数据层”。第 7 层 data rights 判断能不能用，第 6 层 lineage 证明发生了什么，第 2/4/5/8 层负责生产或消费被追踪的 artifact。没有 lineage，synthetic data governance、data quality、incident response、self-improvement 都会缺少可审计证据链。

## 第二十五轮回填：Data Deletion / Unlearning Impact

[[concepts/DataDeletionUnlearningImpact边界]] 将第 7 层数据权利、保留策略、事故处置和供应商协作，接到第 6 层 lineage/eval 与第 2 层 retraining / machine unlearning：

- **Data lifecycle execution**：删除请求、撤回同意、保留期到期、错误数据、版权/隐私风险或事故结论，必须变成可执行的系统动作，而不是只停留在政策文本。
- **Delete is not unlearn**：删除是从 product DB、logs、RAG、memory、eval、training dataset、provider state 等存储中移除或限制数据；unlearning 是处理模型参数或行为中的训练影响。
- **Lineage impact query**：每次删除先查影响面：logs、RAG chunks、embedding、memory、eval、training rows、synthetic derivatives、model versions、provider logs、backup 和 downstream reports。
- **Action plan by store**：不同位置对应 delete、restrict、tombstone、anonymize、unlearn、retrain、replace、suppress 或 legal-hold；输出抑制只能止血，不能当成完成 unlearning 的证据。
- **Verification and audit**：用 data absence、re-ingestion prevention、forgetting quality、membership inference、retain utility、regression eval 和 audit trail 验证闭环。

稳定链路：`deletion trigger → requester identity / scope / basis / exceptions → lineage impact query → delete / restrict / anonymize / unlearn / retrain / replace / suppress / legal-hold plan → owner / due date / provider coordination → data absence + model behavior + utility verification → audit record / card update / prevent re-ingestion`。

这一轮明确：data deletion / unlearning impact 主属第 7 层，不是新的“遗忘层”。第 7 层决定请求、例外、owner、供应商和用户响应；第 6 层提供 lineage 影响面与验证；第 2 层只在模型权重受影响时执行 retraining、model replacement 或 machine unlearning。

## 第二十六轮回填：Human Feedback / Annotation Ops

[[concepts/HumanFeedbackAnnotationOps边界]] 将第 7 层真实反馈和组织运营，接到第 6 层证据质量门与第 2 层 SFT/RLHF/reward model：

- **Human judgment as evidence**：人类反馈不是天然真相；它需要任务定义、rubric、校准、质检、仲裁、lineage 和 downstream validation。
- **Feedback types**：demonstration、preference pair/ranking、critique、correction、gold label、expert review 分别服务 SFT、reward model、eval、RAG、memory、skill 或产品 bug。
- **QA primitives**：gold tasks、overlap、consensus、inter-annotator agreement、reviewer adjudication、rework rate、rubric ambiguity 和 label drift 是稳定质量信号。
- **Routing boundary**：标注结果不自动进入训练；先按用途分流到 eval、reward model、SFT/RLHF、RAG、memory、skill、router、product bug 或 reject。
- **Anti-gaming**：低质量 rubric、泄露 gold set、标注员迎合模型风格或 KPI，会把反馈系统变成 reward hacking 通道。

稳定链路：`model/product/eval gap → task definition + purpose/risk/rights → sample selection + provenance → rubric + examples → annotator onboarding + calibration → labeling/ranking/critique/correction → QA + adjudication → dataset card + lineage + split → route to eval / reward model / SFT-RLHF / RAG / memory / skill / router / product / reject → downstream validation + rubric update`。

这一轮明确：human feedback / annotation ops 主属第 6 层，不是“标注外包层”或单独的 RLHF 层。第 7 层负责 workforce、成本、合同、隐私和真实产品反馈；第 6 层负责把人类判断变成可靠证据；第 2 层只消费通过质量门的训练信号。

## 第二十七轮回填：Evaluation Dataset Lifecycle / Benchmark Governance

[[concepts/EvaluationDatasetLifecycleBenchmarkGovernance边界]] 将第 6 层 eval 数据资产治理，接到第 7 层 release gate、risk owner 和产品决策：

- **Eval asset lifecycle**：eval set、private hold-out、public benchmark、regression suite、red-team set、canary set 和 production shadow set 都是会泄漏、饱和、漂移、被优化、需要版本和退役的质量门资产。
- **Split discipline**：dev/tuning、regression、private hold-out、public benchmark、adversarial、canary、shadow production 不能混用；否则会把 eval 变成训练反馈或 leaderboard gaming。
- **Benchmark governance**：公开 benchmark 只能提供横向参考，不能替代私有产品 eval、lineage、leakage check、production correlation 和 post-market monitoring。
- **Refresh / retirement**：样本泄漏、过时、重复、无区分度、不再相关或 grader 不稳定时，必须 quarantine、refresh、promote 或 retire。
- **Decision binding**：每个 eval set 要绑定 objective、owner 和它控制的决策，例如发布、回滚、provider 选择、router 更新、训练数据准入或事故整改。

稳定链路：`capability/product/risk gap → eval objective + owner + decision → sample sourcing + rights + provenance → split → rubric/grader/metric calibration → versioned dataset card + lineage + leakage controls → CI/release/post-market run → drift/saturation/contamination analysis → refresh/retire → context/tool/RAG/router/training/product fix`。

这一轮明确：evaluation dataset lifecycle / benchmark governance 主属第 6 层，不是新的 benchmark 层。Eval harness 负责运行任务和 grader；benchmark governance 负责 eval 数据资产的来源、split、版本、泄漏、刷新、退役和决策用途。

## 第二十八轮回填：Model Release / Rollback Gate

[[concepts/ModelReleaseRollbackGate边界]] 将第 6 层 eval evidence，接到第 7 层发布、灰度、回滚、迁移和对外沟通：

- **Change gate**：model snapshot、prompt、router、skill、tool、provider、RAG/memory 和 policy 变更，都要有 artifact identity、owner、risk tier、blast radius 和 rollback owner。
- **Evidence binding**：release gate 消费 regression、private hold-out、behavior/safety、latency/cost、tool/data/provider、production contract tests，而不是只看公开 benchmark。
- **Staged rollout**：shadow、internal dogfood、opt-in alpha、canary、A/B、tenant/region ramp 是把离线 eval 接到真实流量的中间层。
- **Rollback readiness**：上线前就要定义 rollback trigger、fallback model/provider、disable flag、credential revoke、prompt restore 和用户沟通路径。
- **Provider drift / deprecation**：托管模型更新、alias 漂移和 deprecation 是供应链风险，应用侧需要 usage audit、release notes monitor、compatibility gate 和 migration deadline。

稳定链路：`change proposal → artifact identity + owner + risk tier → required eval suite → release checkpoint + exception path → staged rollout → online monitoring → promote/hold/rollback/disable/migrate → release notes + audit + postmortem + regression update`。

这一轮明确：model release / rollback gate 主属第 7 层，不是新的 DevOps 顶层。第 6 层负责证明候选是否可靠；第 7 层负责决定真实流量、owner、例外、回滚和迁移。

## 第二十九轮回填：LLM-as-Judge / AI Feedback Calibration

[[concepts/LLMJudgeAIFeedbackCalibration边界]] 将第 6 层 grader / verifier，接到第 2 层 RLAIF / reward model、第 7 层 release evidence 和第 8 层 self-improvement：

- **Judge as instrument**：LLM judge 不是真相源，而是需要 human gold、确定性 checks、bias tests、production outcomes 和 lineage 校准的测量仪器。
- **Hybrid grader stack**：能用 deterministic / code / state grader 时优先不用 LLM judge；开放语义任务才进入 model-based grader，并配 human escalation。
- **Human anchor**：human feedback / annotation ops 生产 gold labels、rubric、adjudication 和 disagreement notes；LLM judge 用它们校准，而不是替代掉整个人类质量系统。
- **Bias suite**：position、verbosity、self-preference、leniency、prompt sensitivity、model-family bias 和 flakiness 必须被单独测量。
- **AI feedback routing**：AI label、AI critique、RLAIF teacher output 和 judge score 先进入候选池；只有通过 quality、rights、lineage、hold-out 和 downstream validation 才能进训练、eval、router、release gate 或 self-improvement。

稳定链路：`task/product/eval objective → choose deterministic/model/human/hybrid grader → rubric + judge prompt + schema → human gold anchor → judge run with versioned model/prompt/order → agreement/calibration/bias/flakiness/product-correlation checks → mitigation + human escalation → route to eval/reward/RLAIF/router/release/self-improvement or reject → monitor judge drift`。

这一轮明确：LLM-as-judge / AI feedback calibration 主属第 6 层，不是新的“AI 评估层”。第 2 层可以消费 AI feedback，但第 6 层先决定它是否是可信证据。

## 第三十轮回填：Eval Result Interpretation / Decision Threshold

[[concepts/EvalResultInterpretationDecisionThreshold边界]] 将第 6 层 eval asset、grader 和 evidence，接到第 7 层 release gate、rollback、canary、SLO 和组织 owner：

- **Score is measurement**：eval score 只是测量值；必须和 baseline、最小有意义差异、uncertainty、risk tier 和产品 SLO 一起解释。
- **Decision first**：每个 eval 先写清楚控制的决策：发布、回滚、路由、训练、自我改进还是继续调查；没有决策对象的 eval 只是观察。
- **Threshold stack**：稳定阈值由 primary metric、guardrails、slice checks、cost/latency、安全约束和 confidence band 组成。
- **Gray zone**：置信区间重叠、样本不足、judge disagreement、flaky task 或关键 slice 低样本时，进入人工复核、专家评审、canary 或补样本。
- **Production correlation**：离线 eval threshold 要持续和真实投诉、人工接管、事故、成本、延迟、满意度和 ROI 对齐。

稳定链路：`decision to control → metric family → baseline and minimum meaningful effect → sampling/repeated trials/slices → uncertainty and flakiness → practical + statistical + safety interpretation → pass/fail/gray-zone/canary/rollback → record owner, exception, production correlation, and threshold refresh rule`。

这一轮明确：eval result interpretation / decision threshold 主属第 6 层，不是新的评估顶层。它把“得了多少分”翻译成“现在该做什么”，再交给第 7 层 release/rollback gate 执行。

## 第三十一轮回填：Production Contract / Compatibility Test

[[concepts/ProductionContractCompatibilityTest边界]] 将第 6 层可测试契约接到第 7 层 provider/model lifecycle、release gate 和 rollback：

- **Contract vs prompt**：prompt 是实现；production contract 是产品必须持续满足的验收标准。
- **Compatibility before replacement**：新模型、provider、prompt、router、skill 或供应链 artifact 进入真实流量前，先和当前 baseline 跑 contract suite。
- **Risk-category suite**：按 schema、任务行为、工具调用、安全、数据处理、风格、成本、延迟和 drift 组织测试，比总体 benchmark 更能发现业务回归。
- **Provider lifecycle input**：release notes、deprecation、recommended replacement、retirement date、redirect 和 silent update 都是 contract test 的触发器。
- **Artifact attestation**：weights、adapter、dataset、container 和 dependency package 的来源、版本、扫描和 release claims 也要进入 compatibility evidence。

稳定链路：`product promise + risk tier + dependency inventory → production contract clauses → compatibility suite → current baseline vs candidate run → threshold/uncertainty interpretation → admit/limit/reject/canary/fallback/rollback → explicit contract update only through product/risk review`。

这一轮明确：production contract / compatibility test 主属第 6 层，不是新的 DevOps 顶层。它定义“仍然兼容”的可测试证据，再由第 7 层发布门执行迁移、灰度、回滚和用户沟通。

## 第三十二轮回填：Judge Drift / Grader Observability

[[concepts/JudgeDriftGraderObservability边界]] 将第 6 层 LLM judge calibration 扩展成长期运行监控：

- **Judge 是测量仪器**：LLM judge、rubric grader、AI annotator 和 judge panel 不是 ground truth，而是会漂移、失真、被优化和被 prompt/rubric 影响的测量系统。
- **Calibration vs observability**：calibration 证明评分器当时和 human anchor 对齐；observability 证明评分器现在仍然和 human anchor、bias suite、production outcome 对齐。
- **Judge health signal**：human agreement、calibration curve、score distribution、flip rate、bias suite、prompt sensitivity、production correlation、latency/cost 和 schema failure 都要进入 dashboard。
- **Score quarantine**：当 judge health 异常时，不应让新分数自动进入 release gate、training feedback、router 或 self-improvement。
- **Judge update as artifact**：judge model、provider、prompt、rubric、schema、examples 和 sampling 的任何变更，都要像模型发布一样有 owner、diff、eval、rollback 和历史分数可比性说明。

稳定链路：`judge inventory → human anchor / canary rerun → score/agreement/bias/production-correlation monitoring → drift diagnosis → recalibrate / freeze / replace / human review / quarantine scores`。

这一轮明确：judge drift / grader observability 主属第 6 层，不是新的顶层。它位于 LLM judge calibration 之后、decision threshold 之前：先判断“评分器是否还可信”，再把可信分数交给阈值和第 7 层 release gate。

## 第三十三轮回填：Intolerable Risk Threshold / Stop Rule

[[concepts/IntolerableRiskThresholdStopRule边界]] 将第 6 层风险证据接到第 7 层组织 hard stop：

- **Risk tolerance 是组织边界**：第 6 层提供 eval、red-team、capability、incident 和 safeguard evidence；第 7 层定义哪些风险 acceptable、tolerable with safeguards、intolerable / prohibited。
- **Hard stop 不能被平均分抵消**：法律禁止、严重能力阈值、safeguard 不足、重大事故、critical guardrail breach 和不可接受 residual risk，不能用总体 benchmark 提升或商业收益抵消。
- **Capability threshold → required safeguards**：frontier safety 框架通常把高危能力阈值和 stronger safeguards、external review、risk report、scope restriction 绑定。
- **Stop rule 是 release gate 的输入**：release gate 执行 pause、block、restrict、rollback、external review；intolerable risk threshold 定义触发条件。
- **Ambiguity 需要 precautionary rule**：高严重度且证据不确定时，默认限制、补证据或外部审查，而不是默认放行。

稳定链路：`risk evidence → risk tolerance / capability threshold → required safeguards / prohibited use / no-go condition → residual risk review → stop, pause, restrict, rollback, external review, or accept with owner`。

这一轮明确：intolerable risk threshold / stop rule 主属第 7 层，不是第 9 层“安全层”。它消费第 6 层证据，约束第 7 层发布、训练、部署、供应商和事故响应，并反哺 eval 与 safeguards。

## 第三十四轮回填：Eval Uncertainty Communication / Release Card

[[concepts/EvalUncertaintyCommunicationReleaseCard边界]] 将第 6 层 eval evidence、uncertainty、limitation 和 judge health，接到第 7 层 release gate、safety governance 和组织 owner：

- **Evidence is not enough**：eval 分数、CI、slice、flakiness 和 grader health 如果只留在实验表里，就不能稳定约束发布决策。
- **Release card 是证据包**：每次 model/prompt/router/tool/RAG/provider 变更，都要把 claim、system under test、baseline、uncertainty、validity scope、limitation、decision band、owner、exception、rollback trigger 和 monitor 写清楚。
- **Uncertainty becomes action**：置信区间重叠、样本不足、judge disagreement、关键 slice 低样本或 benchmark quality 弱，不是“勉强通过”，而是 gray-zone、canary、human review、补样本或 no-go。
- **Card vs gate**：release card 固化证据和局限；release gate 执行发布、回滚、暂停、限制或例外。
- **Card vs model/system card**：model/system card 是模型或系统级透明度文档；release/eval card 是某一次变更或用例的决策文档。

稳定链路：`eval package → uncertainty + validity + limitation → decision band → release/eval card → owner / exception / rollback / monitor → post-release comparison`。

这一轮明确：eval uncertainty communication / release card 主属第 6 层，不是新顶层。它让第 6 层证据不会在进入第 7 层组织决策时丢失不确定性、灰区和边界。

## 第三十五轮回填：Offline / Online Eval Correlation

[[concepts/OfflineOnlineEvalCorrelation边界]] 将第 6 层离线 eval、release card 和线上 production outcome 接成预测校准闭环：

- **Offline eval 是候选筛选器**：它用 regression、hold-out、LLM judge、contract test 和 benchmark 快速淘汰坏候选，但不能独自证明真实用户和业务结果。
- **Online outcome 是现实校准器**：canary、A/B、online evaluator、human review、投诉、incident、成本、延迟和业务指标会反过来验证离线分数是否有预测力。
- **相关性本身要被测量**：directional agreement、rank correlation、calibration curve、false pass/false fail 和 slice agreement，决定某个离线指标能否继续进入 release gate。
- **Production failure 回填 eval**：线上失败 trace 进入 annotation / judge review，再变成 regression eval、release card 字段或 threshold update。
- **指标信任有等级**：高相关指标可作为 gate；中等相关指标只作为 diagnostic；低相关或漂移指标要 quarantine、reweight、refresh 或 retire。

稳定链路：`offline eval delta → release-card hypothesis → canary / A-B / online eval → production outcome → directional agreement / calibration → eval trust tier → dataset / threshold / gate update`。

这一轮明确：offline / online eval correlation 主属第 6 层，不是第 7 层业务指标体系。它让第 6 层 eval 资产持续接受真实生产分布校准，避免离线分数和线上价值脱节。

## 第三十六轮回填：Safety Case / Assurance Case

[[concepts/SafetyCaseAssuranceCase边界]] 将第 6 层多源证据，组织成第 7 层高风险发布论证：

- **Safety case 是论证包**：它不是更多 eval，而是 claims、arguments、evidence、assumptions、limitations、defeaters 和 residual risk acceptance 的结构化组合。
- **适用于高风险 gate**：普通变更可用 release card；高风险能力、frontier deployment、外部审查、重大客户或监管场景需要 safety / assurance case。
- **证据来自多个第 6 层边界**：capability eval、red-team、benchmark governance、judge health、offline/online correlation、production monitor、incident postmortem 和 safeguard assessment。
- **Stop rule 是反证边界**：如果触发 hard no-go、critical safeguard failure 或 unacceptable residual risk，safety case 不能用平均质量提升抵消。
- **动态维护**：模型、prompt、tool、provider、data、用户分布、法律或事故变化后，case 必须 refresh，而不是复用旧批准。

稳定链路：`top claim + context → risk model → subclaims → evidence map → uncertainty / defeaters → independent review → residual risk decision → deployment constraint + refresh trigger`。

这一轮明确：safety case / assurance case 主属第 7 层，不是第 9 层“安全证明层”。它消费第 6 层证据，服务第 7 层发布、限制、外部审查和残余风险接受。

## 第三十七轮回填：Post-release Eval Drift

[[concepts/PostReleaseEvalDrift边界]] 将第 6 层 release evidence 接到第 7 层真实运行中的证据保鲜：

- **发布不是证据终点**：release card、offline eval、threshold、monitor plan 和 safety case 在发布时有效，不代表在生产分布变化后仍有效。
- **Drift 监控的是证据有效性**：prompt/input、response/output、embedding、score、judge、RAG、tool、provider、policy、user segment 和 SLO 变化，都可能让 release evidence stale。
- **不是普通 uptime monitoring**：普通监控看服务是否运行；post-release eval drift 看“支持这次发布的证据还能不能继续支持当前决策”。
- **和 offline/online correlation 串联**：offline/online correlation 验证离线指标是否预测线上 outcome；post-release eval drift 持续监控这种预测关系是否衰减。
- **行动必须预先绑定**：drift alert 要接到 owner、eval refresh、metric quarantine、threshold/card update、canary、rollback、incident 或 safety case refresh。

稳定链路：`release evidence → production baseline → live prompt/response/trace/outcome signals → drift and stale-evidence detection → refresh, quarantine, canary, rollback, incident, or case update → regression/data flywheel`。

这一轮明确：post-release eval drift 主属第 6 层，不是第 9 层“运维监控层”。它让第 6 层 eval 证据在第 7 层真实产品运行中保持可审查、可刷新、可回滚。

## 第三十八轮回填：Eval Portfolio / Metric Hierarchy

[[concepts/EvalPortfolioMetricHierarchy边界]] 将第 6 层分散 eval 组织成可决策证据组合：

- **一个总分不够**：public benchmark、private hold-out、regression、contract test、red-team、human/judge、online outcome 和 SLO 的作用不同，不能被平均成一个分数。
- **指标有角色层级**：hard stop 一票否决，guardrail 保护底线，primary metric 支持主效果，slice/regression 防止长尾和旧 bug 回归，diagnostic 只定位原因。
- **Portfolio 是 release card 的背后结构**：release card 表达证据，metric hierarchy 决定哪些证据进入卡片、谁优先、冲突时怎么行动。
- **Public benchmark 只是探针**：通用榜单可以看能力趋势，但产品 release gate 需要 private product eval、contract、safety、online correlation 和 post-release drift。
- **冲突不能被隐藏**：平均质量提升不能抵消 safety、privacy、tool permission、cost/latency 或关键 slice 破线。

稳定链路：`decision inventory → evidence inventory → metric roles → hierarchy / conflict rule / owner → portfolio run → threshold interpretation → release card / gate / safety case / refresh task`。

这一轮明确：eval portfolio / metric hierarchy 主属第 6 层，不是新顶层。它把第 6 层 eval 资产组织成能服务第 7 层发布、回滚、治理和产品迭代的证据矩阵。

## 第三十九轮回填：Evidence Freshness / Stale Proof

[[concepts/EvidenceFreshnessStaleProof边界]] 将第 6 层 evidence 生命周期接到第 7 层决策有效性：

- **证据也会过期**：eval result、release card、safety case evidence、judge score、monitor、red-team、contract test 和 online outcome，都只在特定 artifact、时间窗口、分布和假设下有效。
- **Freshness 不是 timestamp**：timestamp 只说明何时产生；freshness / stale proof 说明为什么这份证据现在仍能支持当前 claim。
- **失效有两类**：时间到期导致 expired；model/prompt/tool/RAG/judge/provider/policy/用户分布/incident 变化可能直接 invalidated。
- **状态必须可审计**：fresh、stale、expired、superseded、invalidated、unknown 要进入 evidence ledger、release card 和 safety case。
- **Lineage 是底座**：没有 artifact identity 和依赖图，就无法判断哪个变化影响了哪条证据、哪条 claim 和哪个 gate。

稳定链路：`decision claim + evidence artifact → freshness metadata → dependency map → stale triggers → stale proof → use / downgrade / quarantine / refresh / restrict / rollback / case update`。

这一轮明确：evidence freshness / stale proof 主属第 6 层，不是新顶层。它让第 6 层证据进入第 7 层发布、回滚、safety case 和治理决策前，先证明自己没有过期。

## 第四十轮回填：Monitor Ownership / Escalation Ladder

[[concepts/MonitorOwnershipEscalationLadder边界]] 将第 6 层 monitor signal 接到第 7 层组织行动：

- **监控必须有责任链**：每个 monitor、eval breach、stale evidence flag、release-card watch item 和 user report 都需要 owner、backup、SLA、runbook 和关闭条件。
- **告警不是事故，但要能升级为事故**：低风险信号可以 watch 或 ticket；用户影响、guardrail 破线、证据 invalidated、canary 异常要 page on-call；多团队或高风险伤害进入 incident command。
- **Escalation 和 elevation 分开**：escalation 增加资源或响应强度；elevation 抬到 risk owner、leadership、legal/comms、external reviewer 或 regulator。
- **行动权限要预授权**：canary hold、route away、disable tool、fallback model、increase guardrail、human override 和 rollback 不能事故中才讨论。
- **关闭要有证据**：误报解释、修复 diff、eval refresh、release-card update、rollback record、postmortem action item 都是 closure evidence。

稳定链路：`monitor/eval/user-report signal → signal contract → owner assignment → response SLA → escalation/elevation → action → closure evidence`。

这一轮明确：monitor ownership / escalation ladder 主属第 7 层，不是新顶层。它让第 6 层监控和 eval 结果不会停在 dashboard，而是进入 owner、事故、发布门和风险决策。

## 第四十一轮回填：Metric Conflict Resolution

[[concepts/MetricConflictResolution边界]] 将第 6 层多指标证据接成可行动的优先级规则：

- **冲突不能平均掉**：primary metric 上升但 guardrail、slice、online、SLO 或风险指标下降时，不能用一个总分遮住坏信号。
- **先看 metric role**：hard stop、guardrail、primary、slice、online、business、diagnostic 的决策权重不同。
- **Hard stop 最高优先级**：法律禁止、不可容忍风险、critical safety/privacy/security/tool/data breach 不能被 ROI、平均质量或成本节省抵消。
- **Slice 优先于平均值**：关键用户、人群、语言、工具路径或高风险 workflow 退化时，应限制范围、补样本、canary 或 no-go。
- **Offline/online 冲突要校准**：线上不支持离线预测时，先降 trust tier、查混杂、刷新 eval，而不是继续消费旧指标。
- **Override 必须留痕**：谁批准、为什么、多久复审、用什么 monitor 和 rollback trigger 保护，必须写入 release card。

稳定链路：`metric conflict → classify roles → check freshness/uncertainty/trust → precedence rule → action → owner escalation → release card / monitor / regression update`。

这一轮明确：metric conflict resolution 主属第 6 层，不是新顶层。它是 eval portfolio、decision threshold、release card 和 release gate 之间的冲突解释器。

## 第四十二轮回填：Evidence Attestation / Signed Eval Artifact

[[concepts/EvidenceAttestationSignedEvalArtifact边界]] 将第 6 层 evidence 从“可读”推进到“可验证”：

- **证据要绑定 artifact**：eval result、model snapshot、prompt、judge、dataset split、container、tool/skill package 和 release evidence 都要有 subject digest、版本和身份。
- **签名不是全部**：release gate 不能只看 checksum 或签名存在；还要验证 signer identity、producer/builder、predicateType、expected parameters、policy、trust root 和 freshness。
- **Lineage 与 attestation 分工**：lineage 解释证据从哪来、影响什么；attestation 证明这份具体证据和声明没有被替换，并由预期流程产生。
- **Release card 只引用 verified bundle**：手写分数、未知 signer、artifact mismatch、unexpected parameter 或 stale attestation 不能作为放行证据。
- **供应商信任要显式化**：第三方 model hub 或 provider 的 badge、bundle export、trust root 和本地 verification record 必须进入 provider governance。

稳定链路：`artifact digest → attestation statement → signed bundle → policy verification → release card / safety case / release gate`。

这一轮明确：evidence attestation / signed eval artifact 主属第 6 层，不是新顶层。它补上“证据完整性”这一环，让 eval portfolio、release card、safety case 和 release gate 消费的是可验证证据，而不是容易被替换的文件或口头声明。

## 第四十三轮回填：Monitor Tabletop / Runbook Drill

[[concepts/MonitorTabletopRunbookDrill边界]] 将第 7 层责任链从文档推进到演练验证：

- **Owner 要被演练**：monitor owner、on-call、incident commander、Ops、Comms、risk owner、provider POC 和 release owner 不是写在表里就算有效。
- **场景来自第 6 层信号**：eval breach、stale evidence、judge drift、provider silent update、contract regression、signed evidence mismatch、tool misuse 和 data incident 都能变成 inject。
- **Tabletop 与 drill 分层**：tabletop 验证角色、决策和沟通；controlled runbook drill 验证回滚、provider fallback、tool disable、证据保全和 paging 是否能实际执行。
- **演练必须产出改进计划**：after-action report 要有 owner、due date、验收证据、runbook diff、monitor/eval/gate 更新和 re-drill trigger。
- **它检验 safety ops 是否可执行**：没有演练，release card、monitor ownership、incident playbook 和 safety case 容易变成静态文档。

稳定链路：`scenario seed → exercise objective → inject timeline → role handoff → decision/action → after-action report → improvement plan → re-drill`。

这一轮明确：monitor tabletop / runbook drill 主属第 7 层，不是新顶层。它是第 6 层证据和第 7 层行动之间的组织演练验证器。

## 第四十四轮回填：Override Governance / Residual Risk Acceptance

[[concepts/OverrideGovernanceResidualRiskAcceptance边界]] 将第 6 层证据不足、冲突或灰区，接到第 7 层受控例外和残余风险接受：

- **Override 不是绕过质量门**：它把 exception 变成带 owner、authority、expiry、mitigation、monitor、rollback 和审计记录的风险决策。
- **Hard stop 不可 override**：法律禁止、不可容忍风险、critical safeguard 缺失、严重数据/权限/安全破线，先由 [[concepts/IntolerableRiskThresholdStopRule边界]] 判定 no-go。
- **残余风险要绑定证据缺口**：override record 必须说明哪个 eval、slice、freshness、signature、monitor、release card 或 safety case 证据不足。
- **例外必须会过期**：没有 expiry、review date、补证计划和撤销条件的例外，不应进入 release gate。
- **Override debt 要被统计**：重复例外、过期例外、商业压力例外和同一 owner 反复接受的风险，都是治理健康信号。

稳定链路：`exception request -> hard-stop check -> residual-risk record -> approval authority -> expiry + monitor -> review/revoke/close`。

这一轮明确：override governance / residual risk acceptance 主属第 7 层，不是新顶层。它让第 6 层证据不确定时仍能进入组织决策，但不能吞掉 hard stop、release gate、safety case 和 monitor。

## 第四十五轮回填：Verifier Policy / Trust Root Rotation

[[concepts/VerifierPolicyTrustRootRotation边界]] 将 signed evidence 从“可验证”推进到“按当前可信策略通过验证”：

- **Policy 是验收标准**：签名存在不等于可信；verifier 必须检查 expected subject、predicateType、signer/verifier identity、issuer、builder、参数和 freshness。
- **Trust root 不是常量**：Fulcio/Rekor/CT keys、enterprise CA、provider root、TUF metadata 和 verifier keys 都会更新、过期、撤销或被 compromise。
- **Rotation 是生命周期**：安全轮换需要 add new root、overlap、client compatibility、deprecate old root、reverify affected evidence，而不是直接换 PEM。
- **Verification failure 要有动作**：unknown root、expired metadata、unexpected identity 或 policy mismatch 必须 quarantine、rerun、fallback、hold release、rollback 或进入 override review。
- **VSA 是委托验证接口**：可以信任第三方 verifier summary，但必须验证 VSA 签名、verifier id、policy version、subject、resourceUri 和 result。

稳定链路：`evidence bundle -> verifier policy -> trust material -> policy evaluation -> gate decision -> rotation/revocation/reverification`。

这一轮明确：verifier policy / trust root rotation 主属第 6 层，不是新的供应链顶层。它补上 evidence attestation 后的验收策略，使 release card、safety case 和 release gate 消费的不是“有签名的文件”，而是“当前策略下可信的证据”。

## 第四十六轮回填：Evidence Retention / Audit Packet Lifecycle

[[concepts/EvidenceRetentionAuditPacketLifecycle边界]] 将第 6 层 evidence 接到第 7 层长期审计、检索、重验和处置：

- **Evidence packet 不是 dashboard 链接**：release card、eval run、signed bundle、VSA、policy/root version、approval、monitor snapshot 和 rollback plan 要被打包成可检索记录。
- **Retention class 按风险分层**：普通变更、重大 release、frontier safety case、incident、override、legal hold 的保存期限和处置动作不同。
- **保存最小充分可重验集合**：artifact digest、eval config、dataset split、judge version、signed bundle、verifier policy/root、decision owner 和 disposition record 是核心字段。
- **Legal hold 和删除请求要并入生命周期**：到期销毁、匿名化、tombstone、transfer、legal hold 和解除 hold 后的处置都要留痕。
- **Root/policy/provider 变化后要重验**：audit packet 记录当时的 policy/root 版本，并在 verifier policy、trust root 或 provider 变化后标记 reverify。

稳定链路：`decision -> evidence manifest -> packet assembly -> storage tier -> retrieval/reverify -> legal hold/disposition -> destruction/preservation record`。

这一轮明确：evidence retention / audit packet lifecycle 主属第 7 层，不是新顶层。第 6 层产出可信证据，第 7 层保证这些证据几年后仍能找回、读懂、重验、解释和按政策处置。

## 第四十七轮回填：Override Debt / Exception Analytics Dashboard

[[concepts/OverrideDebtExceptionAnalyticsDashboard边界]] 将单条 override / residual-risk record 接到第 7 层组织健康监控：

- **单条例外和例外债务分开**：[[concepts/OverrideGovernanceResidualRiskAcceptance边界]] 决定某次例外能不能批准；override debt dashboard 判断例外是否正在变成组织习惯。
- **Debt 指标必须风险加权**：override rate、active debt、expired active exception、renewal count、repeat owner、risk-weighted exposure、incident conversion、monitor coverage 和 hard-stop near miss 共同描述质量门是否被透支。
- **Dashboard 必须绑定动作**：warn、leadership review、release freeze、restrict scope、deny renewal、policy review 和 backlog 才让指标成为治理机制。
- **重复例外诊断系统设计问题**：missing eval、stale evidence、business pressure、policy ambiguity、provider/tool/data constraint 都要转成 eval、monitor、gate、product 或 provider backlog。
- **Evidence retention 是数据底座**：没有 [[concepts/EvidenceRetentionAuditPacketLifecycle边界]] 保存结构化 override record，就无法计算趋势、续期、过期、复发和事故转化。

稳定链路：`override / exception / residual-risk records -> normalized ledger -> risk-weighted analytics -> threshold action -> gate/eval/policy backlog -> closure evidence`。

这一轮明确：override debt / exception analytics dashboard 主属第 7 层，不是新顶层。它把第 6 层证据缺口和第 7 层例外批准连接成可治理的组织债务信号。

## 第四十八轮回填：Cross-verifier Policy Drift / Conformance Test

[[concepts/CrossVerifierPolicyDriftConformanceTest边界]] 将 signed evidence 的验收从“单个 verifier 通过”推进到“所有执行点一致通过”：

- **Policy 和 conformance 分开**：[[concepts/VerifierPolicyTrustRootRotation边界]] 定义当前可信 policy/root；cross-verifier conformance test 验证 local CLI、CI、release gate、admission controller、runtime guard 和 delegated VSA verifier 是否执行同一语义。
- **漂移不只来自代码**：policy version、trust root、engine version、input normalization、warn/enforce mode、namespace scope、remote policy cache 和 third-party VSA policy 都会导致结果不一致。
- **Golden corpus 是核心资产**：合法、错 issuer、错 subject、错 predicate、expired evidence、stale root、missing dependency、wrong resourceUri、replay 和 scope miss 都要进入样本集。
- **比较结果要超越 allow/deny**：reason code、policy digest、root version、engine version、input digest 和 timestamp 都要记录，否则语义漂移会藏起来。
- **第 7 层动作必须明确**：高风险 false allow mismatch 应 fail closed；其他 mismatch 进入 resync、pin version、root rotation、admission config 修复、release hold 或 override review。

稳定链路：`canonical verifier policy + trust root -> golden corpus -> multi-verifier run -> decision diff -> drift classification -> block/resync/rollback/override-review -> retained conformance report`。

这一轮明确：cross-verifier policy drift / conformance test 主属第 6 层，不是新顶层。它补上 verifier policy 之后的“执行点一致性”，防止 CI、release gate、admission controller 和第三方 verifier 各判各的。

## 第四十九轮回填：Evidence Minimization / Privacy-preserving Audit Packet

[[concepts/EvidenceMinimizationPrivacyPreservingAuditPacket边界]] 把 audit packet 从“保存完整现场”推进到“用最小充分证据证明 claim”：

- **先定义 claim 再收证据**：release、override、incident、safety case 或 external audit 要先写 evidence need statement，再决定哪些字段必要。
- **字段级最小化是核心接口**：user content、tool I/O、trace、prompt、secret、dataset row、judge rationale 要分别标注 raw、redacted、derived、hashed、encrypted、withheld 或 short-retention。
- **可验证不等于全披露**：digest、signed summary、selective-disclosure proof、redacted sample 和 re-run recipe 可以证明关键 claim，同时把原始敏感样本留在 private annex 或不出域。
- **删除也要可审计**：GDPR/SLSA 式删除、expunging 或 anonymization 需要 tombstone/disposition record，记录请求、scope、authority、time 和剩余证据位置。
- **第 6 层证据进入第 7 层前要做隐私化处理**：signed eval、conformance report、release card 和 safety case 都要检查是否过度保留或过度披露。

稳定链路：`audit claim -> purpose/evidence need -> sensitivity classification -> minimum proof set -> redaction/digest/encryption/selective disclosure/private annex -> scoped access -> retention/legal hold/deletion/disposition`。

这一轮明确：evidence minimization / privacy-preserving audit packet 主属第 7 层，不是新顶层。它补上 evidence retention 之后的“证据最小化与隐私披露”边界，防止 audit packet 变成长期敏感数据仓库。

## 第五十轮回填：Release Gate Debt / Gate Bypass Detection

[[concepts/ReleaseGateDebtBypassDetection边界]] 把 release gate 从“有一个发布审批点”推进到“真实生产变更都能证明经过这道门”：

- **Release gate 和 bypass detection 分开**：[[concepts/ModelReleaseRollbackGate边界]] 决定一次变更能不能发布、灰度、回滚；bypass detection 检查生产状态、部署事件、审批、attestation、exception 和 audit log 是否能证明没有绕过。
- **Required release path 是基线**：每类 artifact，包含 model、prompt、router、tool、skill、provider、RAG/memory、policy，都要先定义必须经过哪些 eval、approval、protected environment、attestation 和 rollout step。
- **Evidence join 是核心动作**：把实际 production state 与 release card、approval、protected environment config、signed bundle、exception record、provider notice 和 audit log join；缺证、过期、路径不对先标为 unknown/ungated。
- **Bypass 必须分类**：admin bypass、shadow approval、direct-to-prod write、ungated artifact、stale exception、gate config drift、concurrency/stale deploy、provider silent path 的治理动作不同。
- **Gate debt 要触发行动**：finding 不能只进 dashboard；要能 block、rollback、quarantine、revoke access、补 retroactive review、修 gate config、freeze release 或进入 postmortem。

稳定链路：`required release path -> production change inventory -> evidence join -> bypass classification -> block/rollback/revoke/fix/freeze -> gate debt ledger`。

这一轮明确：release-gate debt / gate bypass detection 主属第 7 层，不是新顶层。它补上 release/rollback gate、override governance 和 override debt dashboard 之间的盲区：不是只统计已记录例外，而是发现那些没有留下结构化记录的隐藏绕过。

## 第五十一轮回填：Dashboard Calibration / Governance Metric Quality

[[concepts/DashboardCalibrationGovernanceMetricQuality边界]] 把 governance dashboard 从“展示指标”推进到“证明指标能支持组织行动”：

- **Dashboard 和 calibration 分开**：[[concepts/OverrideDebtExceptionAnalyticsDashboard边界]] 这类 dashboard 聚合信号；dashboard calibration 判断这些信号的定义、来源、覆盖、freshness、阈值和行动后效果是否可信。
- **先定义 decision inventory**：release freeze、rollback、deny renewal、risk review、owner ticket、budget、backlog 等决策不同，不能用同一个总分驱动。
- **每个指标都要有 definition sheet**：claim、owner、source、query、numerator、denominator、slice、window、lag、freshness、threshold、action 缺一不可。
- **校准看 outcome 而不是看 UI**：指标触发的行动是否减少事故、绕过、过期例外、用户影响或风险暴露；指标改善但真实 outcome 不改善，就是 gaming 或错配。
- **指标要能 quarantine / retire**：policy、product、provider、judge、risk taxonomy 或组织流程变化后，dashboard 指标可能 stale，必须刷新、降级、拆分、合并或退役。

稳定链路：`decision inventory -> metric claim -> metric definition and lineage -> data quality check -> outcome calibration -> trust tier -> threshold/action ladder -> post-action review -> refresh or retire`。

这一轮明确：dashboard calibration / governance metric quality 主属第 7 层，不是新顶层。它补上第 6 层 eval/metric 与第 7 层 dashboard/action 之间的“这数字到底能不能信”边界，防止 governance dashboard 变成指标剧场。

## 第五十二轮回填：Self-Improvement Gate / Change-Risk Budget

[[concepts/SelfImprovementGateChangeRiskBudget边界]] 把第 8 层 self-improvement 从“能提出改动”推进到“改动必须按风险、证据和回滚能力准入”：

- **Proposal 和 apply 分开**：dream、agent self-edit、feedback loop、MOSS 式 source rewrite 或 weight update 只能先生成 proposed diff；能不能应用由 gate 决定。
- **每个 diff 先标 target layer**：memory/context/skill 属第 4 层，tool/harness/workflow 属第 5 层，eval/judge/reward 属第 6 层，release/policy/monitor 属第 7 层，training/weights 属第 2 层。
- **控制面默认更严格**：修改 eval、reward、logs、permission、release gate 或 monitor threshold，会改变“谁来判断改动好不好”，默认需要人工审查和隔离验证。
- **change-risk budget 限制改动速率**：rollback、false promote、incident、override debt 或 gate debt 上升时，自动 self-improvement 的额度下降。
- **低风险不等于无证据**：即使是 memory 去重、context copy 或低影响 skill metadata，也要有 provenance、diff、抽样 eval 和 rollback。

稳定链路：`trace / outcome / failure -> improvement proposal -> target layer classification -> risk score -> change-risk budget -> gate package -> auto-apply / sandbox / canary / human review / reject -> monitor -> rollback or promote`。

这一轮明确：self-improvement gate / change-risk budget 主属第 8 层，不是第九层。它是第 8 层反哺第 2/4/5/6/7 层之前的准入边界，防止“系统会自我改进”变成“系统可以绕过质量门自我改写”。

## 第五十三轮回填：Memory / Skill Governance Drift

[[concepts/MemorySkillGovernanceDrift边界]] 把第 4 层 memory/skill 从“可复用上下文”推进到“长期 operational context 资产治理”：

- **Memory 和 skill 都会改变未来行为**：memory 会影响 agent 的偏好、事实判断和工具选择；skill metadata 会影响 discovery、selection 和 governance，它们不是普通笔记。
- **写入、检索、使用三段都要评估**：sleeper memory poisoning 可能先写入、后检索、再在未来任务中激活；只检查写入不够。
- **共享范围决定准入门槛**：personal memory、project memory、workspace memory、org memory 和 public skill registry 的 owner、permission、review、expiry 不同。
- **Skill metadata 是供应链面**：`description`、trigger、tags、examples 会操纵召回和选择；脚本安全扫描不能替代语义路由回归。
- **退役和 quarantine 是必需动作**：过时 memory、冲突关系、错误 skill、过宽 skill、环境依赖失效和无 owner artifact 都要能隔离、拆分、合并、删除或退役。

稳定链路：`memory / skill artifact inventory -> owner / scope / permission / version labels -> write/update proposal -> provenance/rights/freshness/conflict/security/routing checks -> drift classification -> admit / quarantine / rewrite / split / merge / deprecate / delete / retire -> retrieval/routing regression eval -> self-improvement gate -> audit and periodic review`。

这一轮明确：memory / skill governance drift 主属第 4 层，不是新顶层。它补上第 4 层与第 5/6/7/8 层之间的长期资产治理边界：memory 和 skill 可以持续让 agent 变强，但也可能持续污染 agent。

## 第五十四轮回填：Product Feedback Closure / Actionability

[[concepts/ProductFeedbackClosureActionability边界]] 把第 7 层 data flywheel 从“收集和分流反馈”推进到“反馈必须被行动、验证并关闭”：

- **Feedback collected 不等于 closure**：真实使用信号必须绑定 trace、artifact version、用户/租户、权限、反馈类型和产品 surface，才能被归因和复现。
- **Actionability 是分流后的执行边界**：有效反馈要进入 ignore/observe、eval、RAG/memory、skill/tool、router/serving、training、product、incident/governance 等明确路径。
- **每条行动要有 closure ticket**：owner、SLA、target layer、artifact id、验证方式、rollback/reopen 条件和 no-action rationale 缺一不可。
- **关闭看证据，不看 ticket 状态**：关闭证据可以是 regression eval 通过、RAG retrieval 改善、tool trace 修复、canary 成功、产品指标改善或事故 action item 验证。
- **自动闭环仍受第 8 层 gate 约束**：从反馈自动生成 memory/context/skill/tool/eval/router/training diff 时，必须进入 change-risk budget。

稳定链路：`product event / feedback / trace / incident / business outcome -> rights and purpose check -> triage and attribution -> closure ticket and target route -> downstream artifact update -> validation -> closure evidence or no-action rationale -> dashboard calibration or reopen -> self-improvement gate if automated`。

这一轮明确：product feedback closure / actionability 主属第 7 层，不是新的数据层或治理顶层。它补上 data flywheel、dashboard calibration 和 self-improvement gate 之间的执行缺口：第 7 层反馈只有转成可验证行动，才有资格反哺第 2/3/4/5/6/8 层。

## 层级如何串起来

1. **基础理论 → 模型构建**：Transformer 和规模律解释了为什么“更多数据、更多算力、更合适的架构”能产生更强基础模型。这里回答“能力从哪里来”。
2. **模型构建 → 推理能力**：预训练、指令微调、RLHF、微调和蒸馏把知识、行为偏好和成本约束写入权重。这里回答“模型能做什么、代价是多少”。
3. **推理能力 → 上下文知识**：模型能力不是独立完成任务；真实任务需要把目标、知识、工具说明、历史状态压进有限上下文。这里回答“当下该让模型知道什么”。
4. **上下文知识 → Agent 系统**：当上下文里包含工具、状态和目标，模型就从回答器变成行动器。Agent loop 把“推理”变成“观察、计划、调用工具、检查结果”的循环。
5. **Agent 系统 → 评估可靠性**：Agent 会修改环境，错误会积累，所以必须评估最终状态、trace、工具选择、权限、安全和成本，而不只看单次回答。
6. **评估可靠性 → 产品组织**：第 6 层不仅产出 eval score，还要解释 uncertainty、guardrail、slice、decision threshold、release card、offline/online correlation 和 safety case evidence；只有被这些质量门约束的能力，才能进入产品、工作流、审批边界和组织流程。
7. **产品组织 → 自我改进**：真实使用产生失败案例、偏好、telemetry、数据权利和流程瓶颈；这些信号先经过 product feedback closure 变成有 owner、artifact、验证和关闭证据的行动，再成为下一轮训练、上下文、工具、eval 和产品路线的燃料。
8. **自我改进 → 前面各层**：自我改进不是第九层，而是受控反馈环：它先提出 diff，经 self-improvement gate 按风险预算、证据和回滚能力审查后，才改数据、prompt/context、工具、eval、产品流程，成熟后才可能改模型训练本身。

## Research Loop 合约

后续每轮只研究一个主题，完成后必须回填本页。

单点研究模板：

| 问题 | 要求 |
|---|---|
| 是什么 | 给出一句稳定定义，不追逐营销名词 |
| 解决什么问题 | 说明没有它时系统哪里失败 |
| 主归属 | 在 8 层中选一个主层级 |
| 上下游 | 写清楚输入、输出、被谁使用、反哺谁 |
| 工程实践 | 说明常见实现、工具或流程 |
| 评估方式 | 说明如何判断它有效 |
| 过时风险 | 标出哪些部分可能随模型能力变化而消失 |

默认信源顺序：论文 / 官方文档 / 官方工程博客 / 开源仓库 / 高质量一手访谈 / 行业文章。论坛和社媒只作为线索，不作为高置信来源。

完成标准：一个概念能稳定归类、能解释上下游、能说明实践和评估方式，并且没有重大未解矛盾。

## V1 完成边界

稳定 v1 不要求穷尽所有 AI buzzword，而要求最后几个新主题都能在不新增顶层的情况下归入链路，并且能解释上游、下游、工程做法、评估方式和回填位置。

已补齐 self-improvement gate、memory/skill governance drift 与 product feedback closure，并完成 [[Review: AI知识体系稳定 v1 结构验收]]。结论：8 层链路稳定，当前无需继续默认研究轮；后续只在遇到具体新概念或工程问题时启动单点 research loop。
