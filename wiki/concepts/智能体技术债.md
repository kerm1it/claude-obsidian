# 智能体技术债（Agentic Technical Debt）

AI 辅助编码的特有技术债形式。由于 AI 代码生成速度极快，创始人可能在几周内构建出功能完整但**脆弱、无文档、难以维护**的系统——在没有人工审查或结构性约束的情况下，AI 倾向于生成可用但不可持续的代码。

## 产生机制

```
AI 生成速度快 → 跳过测试/文档/审查
→ 代码库成为"黑箱"
→ 新功能依赖未测试的假设
→ 系统脆弱，难以扩展
```

## 与传统技术债的区别

| 维度 | 传统技术债 | 智能体技术债 |
|------|-----------|-------------|
| 产生原因 | 时间压力下的快捷方式 | AI 生成速度掩盖了质量问题 |
| 积累速度 | 数周/月 | 数天/周 |
| 可见性 | 开发者通常意识到 | 代码"看起来有效"，问题隐藏更深 |
| 核心危险 | 难以修改 | 难以理解 + 难以修改 |

## 防线：CLAUDE.md 作为架构上下文

Anthropic 提出的核心解法是维护 **CLAUDE.md** 文件，作为 AI 代码生成的约束框架：

```markdown
# CLAUDE.md 应包含：
- 产品目标（产品要做什么/不做什么）
- 技术决策（为什么选择这个架构/框架）
- 边界约束（禁止的模式、必须遵守的规范）
- 测试覆盖要求（哪些功能必须有测试）
- 安全基线（默认加密、输入验证等）
```

CLAUDE.md 让 AI 每次生成代码时都在同一个约束框架内工作，而不是在真空中"自由发挥"。

→ 参见 [[concepts/共享领域语言]]（Matt Pocock 的 CONTEXT.md 是同一问题的不同解法）

## 实践建议

1. **每个 PR 补测试**：AI 写完代码，立刻要求 AI 写对应测试
2. **代码审查不能省**：即使是 AI 生成的代码，也需要人工理解
3. **文档同步生成**：每次功能完成立刻更新 CLAUDE.md 和 API 文档
4. **安全扫描前置**：从第一天就运行安全扫描，不要等 Launch 阶段
5. **测试套件即护城河地图**：每发现竞品会出错的 edge case 就加一个测试

## Scale 阶段的代价

Launch 阶段未处理的智能体技术债，在 Scale 阶段会以更高的代价显现：
- 企业采购团队会进行技术尽调
- SOC 2、GDPR、HIPAA 合规要求暴露代码级漏洞
- 多年期企业合同要求 SLA 和可审计性
