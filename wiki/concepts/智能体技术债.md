# 智能体技术债（Agentic Technical Debt）

AI 辅助编码的特有技术债形式。由于 AI 代码生成速度极快，创始人可能在几周内构建出功能完整但**脆弱、无文档、难以维护**的系统——在没有人工审查或结构性约束的情况下，AI 倾向于生成可用但不可持续的代码。

## 产生机制

```
AI 生成速度快 → 跳过测试/文档/审查
→ 代码库成为"黑箱"
→ 新功能依赖未测试的假设
→ 系统脆弱，难以扩展
```

## 与传统技术债的区别

| 维度 | 传统技术债 | 智能体技术债 |
|------|-----------|-------------|
| 产生原因 | 时间压力下的快捷方式 | AI 生成速度掩盖了质量问题 |
| 积累速度 | 数周/月 | 数天/周 |
| 可见性 | 开发者通常意识到 | 代码"看起来有效"，问题隐藏更深 |
| 核心危险 | 难以修改 | 难以理解 + 难以修改 |

## 两道防线：CLAUDE.md + Scope 文档

### 1. CLAUDE.md — 架构上下文（持久记忆）

> "CLAUDE.md files serve as project-level instructions for Claude Code, providing project-specific context and instructions that are automatically read by the Agent SDK when it runs in a directory. Functionally, they are persistent 'memory' for your project."

在打开 Claude Code **之前**，先用 Claude 定义：要遵循的架构模式、要避免的依赖、正在做的权衡及原因。这个输出存为 CLAUDE.md，是构建的第一个产出物，每个后续 session 都依赖它。

```markdown
# CLAUDE.md 应包含：
- 产品目标（产品要做什么/不做什么）
- 技术决策（为什么选择这个架构/框架）
- 边界约束（禁止的模式、必须遵守的规范）
- 测试覆盖要求（哪些功能必须有测试）
- 安全基线（默认加密、输入验证等）
```

**Session 工作流：**
- **开始时**：(1) 重新审视 scope 文档 (2) 加载 CLAUDE.md 架构上下文
- **结束时**：更新 CLAUDE.md，记录构建了什么/做了什么决定/引入了什么假设
- **每次 5 分钟文档 = 对抗架构漂移的便宜保险**

→ 参见 [[concepts/共享领域语言]]（Matt Pocock 的 CONTEXT.md 是同一问题的不同解法）

### 2. Scope 文档 — 功能边界（对抗无摩擦蔓延）

与 CLAUDE.md 并列，还需要一个范围文档，描述：
- 产品**做什么**
- 产品**刻意不做什么**
- 功能修改标准：来自真实用户的什么具体证据才能证明添加新东西是对的

把决策点从"我们应该构建这个吗？"移到"足够多的用户告诉我们没有这个就无法获得价值了吗？"

当新功能想法浮现时，用 Claude 压力测试：是真正来自用户的信号，还是创始人热情伪装成产品思考？

## 实践建议

1. **每个 PR 补测试**：AI 写完代码，立刻要求 AI 写对应测试
2. **代码审查不能省**：即使是 AI 生成的代码，也需要人工理解
3. **文档同步生成**：每次功能完成立刻更新 CLAUDE.md 和 API 文档
4. **安全扫描前置**：从第一天就运行安全扫描，不要等 Launch 阶段
5. **测试套件即护城河地图**：每发现竞品会出错的 edge case 就加一个测试

## Scale 阶段的代价

Launch 阶段未处理的智能体技术债，在 Scale 阶段会以更高的代价显现：
- 企业采购团队会进行技术尽调
- SOC 2、GDPR、HIPAA 合规要求暴露代码级漏洞
- 多年期企业合同要求 SLA 和可审计性
