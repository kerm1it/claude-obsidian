# 固定时间窗实验

## 定义

实验设计约束：每次训练运行使用**固定的挂钟时间**（wall clock time），而非固定步数或 epoch 数，确保不同架构/配置间的实验结果可直接比较。

## Autoresearch 中的实现

- **固定 budget：** 5 分钟（排除启动时间）
- **评估指标：** val_bpb（validation bits-per-byte）
- **吞吐量：** 约 12 次实验 / 小时，过夜约 100 次
- **硬件：** H100（单 GPU）

## 为什么用时间窗而非步数？

| 方案 | 问题 |
|------|------|
| 固定步数 | 不同架构每步计算量不同，大模型慢、小模型快，不公平 |
| 固定 epoch | 数据量固定时，不同 batch size 导致步数不同 |
| **固定时间窗** | 在相同资源约束下比较，反映"真实研究中的机会成本" |

时间窗方案本质上模拟了"研究员能分配的 GPU 时间是固定的"这一现实约束。

## 为什么 val_bpb？

bits-per-byte 是信息论指标，与词表大小无关：
- 换用不同 tokenizer / 词表的模型仍可直接比较
- 比 perplexity 更稳健（perplexity 随词表大小变化）

## 推广应用

固定时间窗的思想可迁移到其他 agent 研究场景：
- 代码优化：固定时间内让 agent 反复 profile → 改 → benchmark
- 超参搜索：替代传统 grid search，让 agent 在时间预算内自由探索
