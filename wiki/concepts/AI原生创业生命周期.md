# AI 原生创业生命周期

Anthropic 在《The Founder's Playbook》中提出的四阶段创业框架，描述 AI 原生创业公司从构想到规模化的完整路径。核心主张：**AI 把季度压缩成周**，验证从月到午，原型从团队到个人，运营从人工到 AI。

## 四阶段概览

```
Idea → MVP → Launch → Scale
```

### Stage 1: Idea（构想）
**目标**：验证 PMF 假设，避免过早执行

**退出三问**（三个都要答"是"才能离开 Idea Stage）：
1. 问题真实且具体吗？（能说清楚谁/多频繁/多严重/现在如何应对）
2. 你的方案真正解决了验证后揭示的问题吗？（不是你最初假设的问题）
3. 信号足够支撑提交 MVP 是有理由的决定，而非信仰行为吗？

**四大陷阱**：
1. **把构建当成验证**：AI 让"想法 → 原型"极快，但原型 ≠ 验证。42% 的创业公司因为构建了无人需要的东西而失败。"原型是压力测试工具，对话才是证据"
2. **过早规模化**：构建太容易，在验证问题-方案契合之前就扩大执行——在不知情中偏离
3. **确认偏见被 AI 放大**：向 AI 要支持性证据，它就会找到。解药：让 Claude 论证为什么竞争对手会成功而你不会
4. **竞争者忽视**：专注自己愿景导致系统性低估竞争对手。解药：让 Claude 分析竞品为何比你更好

**AI 如何帮助**：
- 压力测试问题陈述直到变成可测试假设（"合同审查太慢" → "中市场公司内部法务团队每次合同审查花3天+，因为红线在邮件线程而非版本控制文档中管理"）
- 研究 TAM/SAM/SOM，识别市场是扩张/整合/成熟
- 按层级绘制竞争图（直接/间接/潜在收购方/邻近玩家）
- 识别3个关键外部趋势，评估顺风/逆风
- 设计用户访谈框架（问相关的过去，不问假设性未来）
- **Claude Cowork 自动化访谈运营**：研究潜在受访者列表 → 批量个性化外发邮件 → MCP 接入 Gmail+Calendar 安排访谈 → 定节奏自动 follow-up → 更新跟踪表

**每5次访谈后**：让 Claude Cowork 产出两个列表——支持假设的证据 vs 挑战假设的证据。如果第一个明显更长，问 Claude 这是数据还是你想听的

**退出条件**：用户访谈 + 数据 + 分析共同支撑的 PMF 假设

---

### Stage 2: MVP（最小可用产品）
**目标**：构建真实用户可重复使用的产品，同时不积累拖累扩展的技术债

**退出条件**：特定可识别用户群有证据：留存（回来用）/ 营收（愿意付费）/ 推荐（告诉别人）

**四大挑战**：
1. **智能体技术债** → [[concepts/智能体技术债]]：AI 技术债是复利式的——无约束文件的情况下每个 session 从零推导基本决定，决定漂移，形成无一致心智模型的代码库
2. **虚假 PMF（心理最迷惑）**：早期势头（创始人朋友/HN 头版）≠ PMF。判断方式：Sean Ellis 测试（40%+ "非常失望" = PMF 信号）；努力测试（PMF 后产品开始自己"拉"用户，不再靠创始人推）
3. **无摩擦功能蔓延**：每个单个添加都是可辩护的，但产品会超出边界。传统上工程师时间是对抗蔓延的自然约束，AI 时代这个约束消失了
4. **安全漏洞**：AI 生成的代码有效，但不一定安全。功能代码容易验证，安全漏洞是隐形的直到被利用，没有自然反馈循环。MVP 上线前必须安全审查

**关键实践**：
- **CLAUDE.md 工作流**：打开 Claude Code 之前先用 Claude 定义架构原则/依赖限制/权衡 → 存为 CLAUDE.md → 每次 session 开始时加载 → 每次 session 结束时更新（每次5分钟文档 = 对抗架构漂移的便宜保险）
- **预设量化目标**：在第一个用户出现前定义留存基准、激活标准、Day 7/Day 30 目标、假阳性长什么样（注册但不激活？有收入但无留存？）
- **Claude Cowork 管理反馈循环**：用户外发、反馈 session 安排、Bug 分诊、迭代周期跟踪
- **3轮迭代后诊断**：给 Claude 留存数据+用户反馈+原始假设，问：①数据中有哪个细分群不同？②差距是定位问题还是产品问题？③什么条件必须成立才能有 PMF？

### Stage 3: Launch（发布）
**目标**：把早期牵引力变成可重复、可持续的增长引擎；证明业务值得增长

**退出三条件**：
1. 增长可重复且渠道驱动：CAC/LTV/回收期是你知道且能捍卫的数字
2. 产品能处理生产工作负载：基础设施加固、安全合规到位、可靠性在真实条件下成立
3. 运营无需创始人瓶颈：流程和自动化就位，你不再亲自处理支持/分诊/Sprint规划/报告

**四大挑战**：
1. **技术债来收账**：MVP 代码库为速度构建，生产流量开始暴露快捷方式
2. **创始人成为瓶颈**：信号——决策需要一周才轮到你/支持请求堆积/运营任务靠你记得才发生
3. **过早扩张（会杀死PMF）**：新市场引入新的用户行为/合规要求/支付基础设施——你失去解读自己数据的能力，同时忽视原始用户群
4. **安全合规不可推迟**：有了真实用户和企业合同，漏洞变成真实暴露风险

**三工具协同（Launch Stage 精髓）**：
> "When Claude Code builds the product, Claude Cowork builds the company around it, and Claude helps operationalize this product and organizational knowledge, a small team can run like a company N× its size."

**运营 OS**（Claude Cowork 实现）：
- Sprint 节奏（scheduling sprint ceremonies）
- 最小规格模板（minimum spec template）
- Bug 分类决策树（bug triage decision tree）
- 周报（weekly metrics brief，从真实数据源自动拉取）

---

### Stage 4: Scale（规模化）
**目标**：建立系统性增长，通过外部审查；创始人从构建者→对外执行官

**退出条件（三选一）**：
1. 不依赖外部资本的可持续盈利
2. IPO 就绪
3. 被收购

三种情况都要求：增长系统性且可审计 + 产品护城河经得起审查 + 组织运营成熟且可持续

**创始人保留清单**（其余全部候选委托）：
- 产品叙事决策
- 董事会关系
- 企业级大客户谈判
- 创始人对创始人的对话
- **瓶颈测试**：让 Claude 地图化你的工作流，问每个流程"你一周不在时会怎样？"——停摆的就是仍需加固的委托节点

**GTM 功能构建**：
- Claude 构建：市场细分/消息架构/分析师关系策略/销售剧本/面向投资者的指标叙事（每个受众有自己的词汇和评估标准）
- Claude Cowork 执行：内容管道/外发序列/分析师简报后勤/PR 节奏/CRM 卫生/管道报告
- Claude Code 构建：交互式演示环境/集成文档/沙盒租户/API 参考/技术单页（规模阶段 Loom 视频+销售 Deck 不再足够）

**护城河构建**：
- [[concepts/数据飞轮复利|数据飞轮]]：行为信号 → 产品改进 → 更多使用（时间锁定，竞争对手无法复制）。"Your test suite becomes a map of your moat."
- [[concepts/工作流锁定|工作流锁定]]：深度集成让切换从产品决策变为运营项目
- **领域专业知识 → AI 上下文复利**：通过对话/项目/记忆把行业术语/监管陷阱/edge case 告诉 Claude → 编码进 Skills → 通用 AI 无法匹配

## 压缩效应

| 传统路径 | AI 时代路径 |
|----------|-------------|
| 验证：数月 | 验证：一个下午 |
| 原型：需联合创始人 + 团队 | 原型：清晰的问题 + 几次编码会话 |
| Launch 就绪：launch 前冲刺 | Launch 就绪：持续工作流 |
| 运营：早期雇员救火 | 运营：AI 处理，团队专注判断 |

## 核心结论

> "The bottlenecks are no longer what you can build, but what you choose to build."
