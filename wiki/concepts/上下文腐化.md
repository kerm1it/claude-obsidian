# 上下文腐化（Context Rot）

**来源：** [[entities/Anthropic]]  
**参见：** [[sources/anthropic-context-engineering-agents-2026-05-26]]

---

## 定义

> **上下文腐化**：随着输入 token 数量增加，LLM 性能逐渐退化的现象。

上下文窗口越大，模型并不必然越好。超过某个临界点，增加更多 token 反而**损害**推理质量。

---

## 根本原因：Transformer 注意力的二次复杂度

Transformer 架构中，自注意力机制对序列中每个 token 与其他每个 token 都建立关系：

```
序列长度 n  →  注意力关系数 ≈ n²
```

随着上下文增长：
- 模型需要在更多 token 对之间分配注意力权重
- 信号被噪声稀释，真正相关的信息难以凸显
- 推理深度和准确性下降

---

## 表现形式

| 症状 | 描述 |
|---|---|
| 遗忘早期关键信息 | 远距离依赖丢失（"Lost in the Middle" 现象） |
| 指令漂移 | 随对话延长，对系统提示的遵循度下降 |
| 幻觉增加 | 用错误的模式填补注意力稀释导致的"空白" |
| 一致性破坏 | 同一任务中前后矛盾的推理 |

---

## 为什么扩大窗口不能解决问题

业界常见误解：模型窗口从 8K → 200K → 1M，Context Rot 就不重要了。

事实恰相反：
- 更大的窗口允许用户**塞入更多内容**，加速 Context Rot
- 架构层面的二次复杂度仍然存在（KV Cache 等技术缓解计算成本，但不改变注意力分布规律）
- 解决方案是**主动管理 token 质量**，而非被动依赖窗口扩张

---

## 应对策略

| 策略 | 机制 |
|---|---|
| Compaction | 周期性压缩对话历史，重启干净上下文 |
| 即时检索（JIT） | 不预加载，按需动态拉取 |
| 结构化笔记 | 将记忆外化到上下文窗口之外 |
| 子 Agent 架构 | 每个 Agent 持有聚焦的干净上下文 |
| 工具结果清除 | 任务完成后清除 stale tool outputs |

→ 完整策略体系见 [[concepts/上下文工程]]
