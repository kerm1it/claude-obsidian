# 记忆自动拉取（Auto-fetch）

[[entities/OpenHuman]] 的核心背景机制：**每 20 分钟**自动遍历所有已激活的第三方连接，将最新数据拉入本地 [[concepts/MemoryTree|Memory Tree]]。

## 机制

```
每 20 分钟触发
    ↓
遍历所有活跃连接
（Gmail / Calendar / GitHub / Notion / Slack / ...）
    ↓
拉取最新数据
    ↓
经 [[concepts/TokenJuice]] 压缩
（HTML→MD、URL 缩短、非 ASCII 移除）
    ↓
规范化为 ≤3k-token Markdown 块
    ↓
折叠入 Memory Tree（SQLite + Obsidian .md 文件）
```

## 核心价值

> "The agent already has tomorrow's context this morning."

- **无需 polling 循环**：用户不需要写任何轮询代码，连接即生效
- **无训练期**：一次同步后，agent 就有了邮件、日历、代码仓库、文档的完整（压缩）上下文
- **被动积累**：用户不在线时，数据照常刷新

## 实现背景

传统 agent harness 的冷启动问题：

| Agent | 获取上下文的方式 | 问题 |
|-------|-----------------|------|
| Hermes Agent | 观察用户工作行为 | 需要数周积累 |
| OpenClaw | 等待插件主动传入 | 需要手动配置 |
| Claude Cowork | 手动 MCP 接入 | 需要用户发起 |
| **OpenHuman** | **Auto-fetch 20分钟循环** | 几分钟内生效 |

## 与 Claude Cowork 的对比

[[entities/ClaudeCowork]] 通过 MCP 接入 Gmail/Calendar，但仍需用户在会话中主动触发。Auto-fetch 是**被动、持续、无感知**的——agent 在用户睡觉时仍在更新上下文。
