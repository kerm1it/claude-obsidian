# 软件工厂（AI Software Factory）

**来源**：[[sources/yc-diana-ai-company-ground-up-2026-05-22]]
**提出者**：[[people/Diana]]（YC 合伙人）

---

## 核心思想

> "Humans write a spec and a set of tests that define success. AI agents generate the implementation and iterate until the tests pass."

TDD（测试驱动开发）的下一个演进：**人定义"什么"，AI 决定"怎么"**。

---

## 工作流程

```
人类写 spec（定义功能目标）
    ↓
人类写测试集（定义成功标准）
    ↓
AI Agent 生成实现代码
    ↓
运行测试 → 失败 → AI 迭代修改
    ↓
达到概率满足阈值 → 完成
    ↓
人类判断输出质量（不写代码）
```

---

## 极端案例：Strong DM

- 目标：消除人类写代码和审查代码的需求
- 实现：specs + 场景验证（scenario-based validations）驱动 Agent 写测试和代码
- 结果：**仓库里零手写代码**，只有 specs 和 test harnesses

---

## 人类角色的转变

| 旧世界 | 软件工厂 |
|--------|---------|
| 写代码 | 写 spec |
| Code review | 判断输出质量 |
| 调试 | 定义测试用例 |
| 实现细节决策 | 架构和目标决策 |

---

## 与千倍工程师的关系

软件工厂是实现"千倍工程师"的机制：
- 单工程师 + Agent 系统 = 过去整个工程团队
- 参见：[[concepts/软件民主化]]、[[concepts/ProductOverhang]]

---

## 在 AI 知识体系中的位置

软件工厂是第 7 层产品组织接口中的工程组织形态：第 5 层 agent harness 执行代码生成和测试循环，第 6 层测试/eval 定义质量门，第 7 层把人类角色改写为 spec、验收和责任边界。

稳定链路：`业务目标 → spec → tests / eval → agent implementation loop → review / deployment → telemetry → 下一轮 spec/eval`。

风险不在于 AI 会不会写代码，而在于 spec、测试和组织责任是否足够清楚。否则第 7 层会把第 5 层的速度放大成技术债。
