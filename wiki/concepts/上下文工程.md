# 上下文工程（Context Engineering）

**来源：** [[entities/Anthropic]]  
**参见：** [[sources/anthropic-context-engineering-agents-2026-05-26]]

---

## 核心定义

> **上下文工程**：在 LLM 推理期间，策划并维护最优 token 集合的技术实践。

与**提示工程**（Prompt Engineering）的区别：

| 维度 | 提示工程 | 上下文工程 |
|---|---|---|
| 范围 | 单次 prompt 措辞与组织 | 多轮推理中全部信息的管理 |
| 内容 | 指令文本 | 系统提示 + 工具定义 + 外部数据 + 消息历史 |
| 时间维度 | 静态（写一次） | 动态（随任务持续演进） |

上下文工程是提示工程的**超集**：在管理单次 prompt 措辞的同时，还要管理跨越多个推理步骤的全部信息流。

---

## 为什么重要

LLM 的性能不仅取决于模型能力，还取决于**在正确时刻提供正确信息**。

两种常见失效：
- **信息过少**：Agent 缺乏完成任务所需的上下文，反复询问或重复犯错
- **信息过多**：上下文窗口被冗余内容填满，[[concepts/上下文腐化]] 导致性能下滑

---

## 核心原则

### 1. 高信号 Token 原则

> "找到使期望结果概率最大化的最小高信号 token 集合。"

上下文是有限稀缺资源——即便窗口扩大也应如此对待。每个 token 都应服务于任务目标。

### 2. 系统提示"恰当海拔"

系统提示的抽象层级需要平衡：
- **太具体**：硬编码逻辑，脆弱，无法泛化
- **太抽象**：Agent 缺乏足够引导，在模糊场景中失措
- **恰当**：提供具体约束，保留足够灵活性应对变化

### 3. 工具设计三要素

| 要素 | 含义 |
|---|---|
| 自包含（self-contained） | 不依赖外部隐式状态 |
| 容错（robust to error） | 异常输入有合理降级 |
| 意图明确（clear） | 功能不与其他工具重叠 |

工具集过于臃肿或功能重叠，会让 Agent 产生选择歧义，浪费上下文。

---

## 长任务上下文策略

### Compaction（压缩）

当对话历史超过阈值，将其**摘要压缩**后以新上下文窗口重初始化。保留关键决策与状态，丢弃冗余中间输出。适用于长时 Agentic 任务（代码库改写、多步研究等）。

### 结构化笔记（Scratchpad）

Agent 在上下文窗口**外**维护持久笔记，记录任务进展、关键发现、待完成步骤。将"记忆"外化，减少上下文内的重复信息。

### 子 Agent 架构

专化 Agent 以**干净上下文**处理聚焦子任务，仅将压缩摘要返回给协调 Agent。实现关注点分离，避免单一上下文积累全部任务历史。

### 即时检索（JIT Retrieval）

维护轻量标识符，通过工具**按需动态加载**数据，而非提前注入全部可能相关内容。镜像人类工作记忆：只在需要时才"回忆"。

→ 与 [[concepts/渐进式上下文注入]] 高度相关

---

## 混合方案

没有通用最优策略。最佳实践通常**组合多种方法**：
- 已知需要的数据 → 提前检索（速度优先）
- 未知探索性数据 → JIT 自主检索（灵活优先）
- 长历史 → Compaction 压缩

---

## 实践建议

1. **从最小 prompt 开始**，搭配最强模型
2. **按失败模式迭代**添加指令（而非预先过度指定）
3. 持续把上下文视为**稀缺资源**
