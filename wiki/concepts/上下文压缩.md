# 上下文压缩

在 AI Agent 工具输出、日志、RAG 结果、文件和对话历史到达 LLM 之前进行内容压缩，以节省 token 同时保持答案质量的技术实践。Headroom 是该领域最完整的开源实现。

## 核心原理

[[concepts/上下文工程]] 的"高信号 token"原则的工具化实现：上下文是有限稀缺资源 → 找到使期望结果概率最大化的**最小高信号 token 集合**。上下文压缩是主动管理 token 质量的一种手段。

## Headroom 的六种压缩策略

| 策略 | 对象 | 方法 |
|------|------|------|
| SmartCrusher | JSON | 通用结构化数据压缩 |
| CodeCompressor | 代码 | AST 感知（Python/JS/Go/Rust/Java/C++） |
| Kompress-base | 文本 | HF 模型，agentic traces 训练 |
| CacheAligner | 前缀 | 稳定前缀 → KV cache 命中 |
| IntelligentContext | 上下文窗口 | 基于分数的上下文拟合 |
| CCR | 全部 | 可逆——原文不删，LLM 按需取回 |

## 可逆性（CCR）

关键区分器：CCR（Compressed-Content Retrieval）保留原文在本地存储中，LLM 可通过 `headroom_retrieve` 工具按需取回完整内容。这解决了传统压缩的"信息丢失"问题。

## 与相关概念的层次

```
上下文工程（管理 token 质量的原则框架）
  ├─ 上下文腐化（问题诊断：token 越多性能越低）
  ├─ 渐进式上下文注入 / JIT 检索（策略：按需加载，不预装）
  └─ 上下文压缩（策略：压缩已加载内容，提高信息密度）
      └─ Headroom（工具化实现）
```

## 跨 Agent 记忆

Headroom 提供跨 Claude/Codex/Gemini 的共享压缩存储，自动去重。这是 **Memory + Compression** 的融合——压缩后的记忆比原始记忆更高效。

## headroom learn ↔ Dynamic Workflows

`headroom learn` 挖掘失败 session → 写回 CLAUDE.md 的能力与 Dynamic Workflows 的 Memory and rule adherence 模式独立收敛——都是通过分析历史交互来改进未来行为。
