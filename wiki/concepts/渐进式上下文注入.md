# 渐进式上下文注入（Progressive Disclosure）

**来源：** [[entities/ClaudeMem]]（thedotmack/claude-mem）  
**参见：** [[sources/claude-mem-2026-05-13]]

---

## 核心思想

> 不要一次性把所有历史记忆塞入上下文窗口——分层、按需、带 token 成本可视化地注入。

AI Agent 的记忆注入面临两种极端：
- **注入太少**：Agent 失去历史上下文，重复犯错
- **注入太多**：上下文窗口被历史记录填满，有效工作空间压缩

渐进式上下文注入的解决方案：**先看索引，再看详情，只拉取真正需要的内容**。

---

## 三层结构（claude-mem 实现）

```
Layer 1: search(query)
  → 紧凑索引：ID + 标题 + 摘要（~50-100 tokens/条）
  → 扫描全局，找到候选集

Layer 2: timeline(observation_id)
  → 时间线上下文：该观察前后发生了什么
  → 理解背景，二次过滤

Layer 3: get_observations(ids=[...])
  → 完整详情（~500-1,000 tokens/条）
  → 仅对最终确认相关的 ID 拉取
```

**实测效果：节省约 10x tokens**

---

## 类比

类比于数据库查询优化：
- Layer 1 ≈ `SELECT id, title FROM observations WHERE ...`（索引扫描）
- Layer 2 ≈ 查看相关行的部分字段
- Layer 3 ≈ `SELECT * FROM observations WHERE id IN (...)`（精确拉取）

先过滤，再拉取全量——避免全表扫描式记忆注入。

## Anthropic 的 JIT 检索表述（2026-05-26 更新）

Anthropic 工程博客 [[sources/anthropic-context-engineering-agents-2026-05-26]] 在[[concepts/上下文工程]]框架中独立提出了相同思路，称之为"**即时检索（Just-In-Time Retrieval）**"：

> 维护轻量标识符，通过工具动态加载数据，而非预先处理全部内容。

与 claude-mem 三层结构的对应关系：

| Anthropic JIT 检索 | claude-mem 渐进式注入 |
|---|---|
| 维护轻量标识符 | Layer 1 索引（50-100 tokens/条） |
| 工具调用动态加载 | Layer 3 get_observations（按需拉取） |
| 镜像人类工作记忆 | "先看索引，再看详情"类比 |

两者的侧重点不同：
- **claude-mem**：具体的三层 API 实现（search → timeline → get_observations）
- **Anthropic**：Agent 工具调用层面的通用设计模式，与 Compaction、子 Agent 架构并列

JIT 检索是 [[concepts/上下文工程]] 中应对 [[concepts/上下文腐化]] 的核心策略之一。
