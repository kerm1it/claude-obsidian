# 代码仓库即记录系统（Repo as System of Record）

**来源**：[[sources/harness-engineering-openai-2026-02-11]]，[[people/RyanLopopolo]]（OpenAI）

---

## 核心思想

智能体在运行时无法访问的信息等于不存在。因此，**代码仓库必须成为所有知识的唯一来源**——不只是代码，还包括架构决策、产品规范、团队规范、品味偏好。

> "那次让团队在架构模式上达成一致的 Slack 讨论？如果智能体无法发现它，那么它就会像迟了三个月入职的新员工一样，对其一无所知。"

---

## AGENTS.md 的正确用法

### 错误模式：大型百科全书
| 问题 | 说明 |
|------|------|
| 情境稀缺 | 大文件挤掉任务、代码和关键信息 |
| 指导失效 | 一切都"重要"时，一切都不重要 |
| 立即腐烂 | 无法机械验证，悄然成为陈旧规则坟场 |
| 难以核实 | 不适合覆盖率/新鲜度/所有权检查 |

### 正确模式：目录（~100 行）
- AGENTS.md 只作地图，指向 docs/ 目录下更深层的真实信息来源
- **渐进式披露**：智能体从小而稳定的切入点开始，被指导下一步去哪里

---

## 推荐目录结构

```
AGENTS.md          ← 地图，~100行
ARCHITECTURE.md    ← 域和包分层顶层地图
docs/
├── design-docs/   ← 设计文档，含验证状态和核心理念
├── exec-plans/    ← 执行计划（active/completed/tech-debt）
├── generated/     ← 自动生成文档（db-schema等）
├── product-specs/ ← 产品规范
├── references/    ← 第三方库 llms.txt
├── DESIGN.md
├── FRONTEND.md
├── PLANS.md
├── PRODUCT_SENSE.md
├── QUALITY_SCORE.md  ← 质量评分，追踪差距
├── RELIABILITY.md
└── SECURITY.md
```

---

## 机械验证

- 专职 linter 验证文档新鲜度、交叉链接、结构正确性
- CI 作业持续检查
- `doc-gardening` 智能体定期扫描过时文档，发起修复 PR
